## Fix applied in this plan

### Task 1a ŌĆö Migration idempotency (primary fix)

**Symptom (pre-fix, flagged in 13.2-01 + 13.2-03 summaries):**
```
SQLSTATE[42P07]: Duplicate table: 7 ERROR:
relation "finngen_analysis_modules_pkey" already exists in schema "finngen"
(SQL: ALTER TABLE app.finngen_analysis_modules SET SCHEMA finngen)
at database/migrations/2026_04_19_000100_isolate_finngen_schema.php:65
```

**Root cause:**
`migrate:fresh` on `parthenon_testing` calls `db:wipe`, which drops tables
only in the `pgsql_testing` connection's `search_path` = `app,php,public`.
The `finngen` schema and its tables (left behind by a prior successful run
of the 13.1 `isolate_finngen_schema` migration) survive the wipe. When the
CREATE TABLE migrations at 2026_04_13_* then re-create `app.finngen_*`,
and the 13.1 `isolate_finngen_schema` re-runs, its `ALTER TABLE ... SET
SCHEMA finngen` tries to move each table into `finngen`, carrying the
original pkey names. The target schema already contains indexes with those
same pkey names from the prior run, triggering 42P07 on the very first
SET SCHEMA statement (L65).

**Fix (minimal, idempotent):**
Added a "Block 1b" to the migration's `up()` method, between `CREATE SCHEMA
IF NOT EXISTS finngen` and the SET SCHEMA block, that issues
`DROP TABLE IF EXISTS finngen.{T} CASCADE` for each of the 7 target tables.
On a clean DB this is a no-op (nothing to drop). On a re-entry parthenon_testing
migrate:fresh cycle it clears the stale residue so Blocks 2-4 proceed cleanly.

**Verified effect:**
- `tests/Feature/FinnGen/IsolationMigrationTest` ŌĆö 4/4 tests PASS (was 4/4 FAIL)
- `tests/Feature/FinnGen/FinnGenSchemaExistsInTestDbTest` ŌĆö 1/1 PASS
- All 30 migrations apply cleanly on `parthenon_testing` via `RefreshDatabase`
- `migrate:fresh` can be re-run arbitrarily without hand-intervention

## Architectural Blocker ŌĆö Out of 13.2-04 Scope

### The cross-connection transaction isolation problem

All 41 `finngen_runs_user_id_fkey` violations + 3 endpoint_definitions_pkey
violations + 8 cascaded HTTP 500s (~52 of the 71 failures, ~73%) stem from
a single pre-existing architectural gap in the FinnGen test harness.

**Root cause:**
1. `backend/config/database.php` defines a `finngen` connection hardcoded to
   `env('DB_DATABASE', 'parthenon')` ŌĆö it points to the **live DEV database**
   even in the testing environment (`APP_ENV=testing`), because `phpunit.xml`
   only overrides `DB_TEST_DATABASE` and `DB_CONNECTION`, not `DB_DATABASE`.
2. All FinnGen models (`App\Models\App\FinnGen\Run`,
   `FinnGenEndpointGeneration`, `EndpointDefinition`, `AnalysisModule`,
   `WorkbenchSession`, `FinnGenUnmappedCode`) hardcode
   `protected $connection = 'finngen'`.
3. When a test seeds a user via the default `pgsql_testing` connection
   (inside a RefreshDatabase transaction), then creates a `Run` via the
   `finngen` connection, the Run INSERT is sent to the **wrong database**
   (live `parthenon`), where the just-seeded user does not exist ŌåÆ FK
   violation on `finngen_runs_user_id_fkey`.
4. When the finngen connection happens to target the right DB (via env
   override), RefreshDatabase only wraps `pgsql_testing` in a transaction,
   so writes via the `finngen` connection are never rolled back between
   tests ŌĆö causing the observed unique-constraint violations on the second
   test case (state leak).

**Runtime confirmation (tinker):**
```
finngen.database = parthenon          ŌåÉ SHOULD be parthenon_testing
pgsql_testing.database = parthenon_testing
```

**Why it's pre-existing:**
The gap was opened when the `finngen` connection was introduced in Phase 13.1
Plan 01 (`af5d10e4f feat(13.1-01): add finngen + finngen_ro Laravel connections`).
Phase 13.1's scope boundary in 13.1-05-SUMMARY ┬¦SC-10 explicitly deferred
"parthenon_testing bootstrap green" to a later phase. Phase 13.2 planning
focused on 4 cutover closures (R worker + controller + grants + Pest run)
but did not include a plan to add a `finngen_testing` connection + model
overrides. No prior phase has solved this.

**Why 13.2-04 cannot fix it:**
The required fix involves:
- Adding a new `finngen_testing` connection in `backend/config/database.php`
  (mirroring the `inpatient_testing` precedent used by Morpheus).
- Changing 6 FinnGen model `$connection` fields to read from
  `config('finngen.connection', 'finngen')` with an override set via
  phpunit.xml `FINNGEN_DB_CONNECTION=finngen_testing`.
- Adding `$connectionsToTransact = ['pgsql_testing', 'finngen_testing']`
  to `tests/TestCase.php` (or a new FinnGenTestCase base class).
- OR setting `DB_DATABASE=parthenon_testing` + `DB_USERNAME=smudoshi` +
  `DB_PASSWORD=acumenus` in phpunit.xml (simpler but breaks vocab-connection
  permission model under `parthenon_app` default).

All three options modify Wave 1/2/3 files (models) or add a new connection
infrastructure, which the 13.2-04 plan's `<parallel_execution>` scope
explicitly forbids:
> FORBIDDEN: Do NOT modify any code from Waves 1/2/3 (controller, R worker,
> models). Your job is the parthenon_testing environment ŌĆö migrations +
> test fixtures only.

Escalation required: this belongs to a new follow-on plan (proposed
13.2-06 or folded into 13.3).

## Secondary Blocker ŌĆö Missing USAGE grant on `app` schema for `parthenon_migrator`

**Symptom:**
```
SQLSTATE[42501]: Insufficient privilege: 7 ERROR: permission denied for schema app
LINE 1: SELECT 1 FROM ONLY "app"."cohort_definitions" x WHERE $1 OPE...
QUERY:  ... FOR KEY SHARE OF x  (Connection: pgsql_testing, SQL: delete from "users" where "email" = admin@acumenus.net)
```

**Root cause:**
`has_schema_privilege('parthenon_migrator', 'app', 'USAGE')` returns
**false** on `parthenon_testing` but **true** on `parthenon`. After Plan 02
transferred ownership of `app.cohort_definitions` to `parthenon_migrator`,
PostgreSQL's referential-integrity trigger runs the FK-validation SELECT
in the owner's privilege context; `parthenon_migrator` can't see the
schema it owns a table in. Affects any test that deletes an admin@acumenus
user (triggering FK on cohort_definitions.author_id).

**Why it's pre-existing:**
Plan 13.2-02's role-split codification migration transfers ownership but
does NOT codify the USAGE grant to `parthenon_migrator` on `app`. The
grant exists in DEV `parthenon` (manually applied during Phase 13.1's
role-split window per `13.1-DEPLOY-LOG.md ┬¦Deviations D4/D5/D6`) but was
never codified into the Plan 02 migration. Another gap for the follow-on
plan.

**Why 13.2-04 cannot fix it:**
Adding a `GRANT USAGE ON SCHEMA app TO parthenon_migrator` statement
requires either (a) modifying Plan 02's migration (**forbidden** per the
scope directive "Do NOT modify the 13.2 migrations (Plans 01 + 02)"),
or (b) adding a new migration file (out of scope for a migration-idempotency
+ test-fixture plan).
