## Residual Blocker — Out of Plan 13.2-06 Scope

### Cross-connection foreign key visibility

**43 of the 62 remaining failures** (~70%) trace to a single Laravel
architectural constraint:

```
SQLSTATE[23503]: Foreign key violation: 7 ERROR:
insert or update on table "runs" violates foreign key constraint
"finngen_runs_user_id_fkey"
DETAIL: Key (user_id)=(479) is not present in table "users".
(Connection: finngen_testing,
 SQL: insert into "runs" ("user_id", ...) values (479, ...))
```

**Root cause:**

1. Tests seed a User via the default `pgsql_testing` connection (wrapped
   in a transaction by RefreshDatabase).
2. Tests then create a `Run` via `Run::create()`, which (after this
   plan's Task 2) routes through the `finngen_testing` connection —
   a **separate PDO** in a **separate transaction**.
3. PostgreSQL's FK trigger on `finngen.runs.user_id → app.users.id`
   runs WITHIN the finngen_testing transaction's snapshot. The User
   inserted via pgsql_testing's transaction is **not visible** —
   hence the 23503 violation.

**Why 13.2-06 cannot close this gap:**

All three of the viable fixes expand scope beyond this plan's charter:

| Option | Cost | Concern |
|---|---|---|
| **A. Shared-PDO override** — override Laravel's ConnectionResolver so `finngen_testing` reuses `pgsql_testing`'s PDO | ~50 LOC in a test bootstrap | Subtle; potential side effects on concurrent-query semantics; needs review |
| **B. DatabaseTruncation instead of RefreshDatabase** — truncate tables between tests instead of transaction-wrap | Touch ~30+ test files | Large behavioral change; slower runs (2-3x); different test semantics for cache-backed state |
| **C. Commit user-fixture outside RefreshDatabase** — seed a fixture user into parthenon_testing once before suite, run tests with that user; no per-test User::factory | Touch dozens of test setUps | High churn; rejects the "RefreshDatabase per-test isolation" ethos |

**Why it surfaced AFTER Task 1+2 closed the primary gap:**

The 13.2-04 PEST-LOG correctly identified that the `finngen` connection
was pointing at the WRONG DB (live DEV parthenon). Closing that gap
(Task 1+2 of this plan) re-routed the writes to `parthenon_testing` ✅,
but revealed a SECOND layer: even with both connections correctly
pointing at `parthenon_testing`, the separate-PDO-per-Laravel-connection
model still prevents the FK check from seeing the cross-connection
User INSERT when both are inside un-committed transactions.

**Recommendation:** follow-on plan 13.2-07 (or fold into 13.3) to
implement Option A. Estimated LOC: ~100 (test bootstrap override +
smoke validation). Would close all 43 remaining FK failures +
3 cascading 500s + 14 assertion cascades = ~60 of the 62 residual
failures.
