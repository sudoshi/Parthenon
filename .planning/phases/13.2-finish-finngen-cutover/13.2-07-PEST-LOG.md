## Critical deviations from the plan

The plan's task body assumed two facts that proved incorrect under Laravel 11's
actual source, and a third that surfaced during Pest execution:

### Deviation 1 — `init<TraitName>` does NOT auto-invoke (Rule 1: bug)

The plan said:

> Trait methods with the name `init<TraitName>` are auto-invoked by
> Laravel's `InteractsWithTestCaseLifecycle` machinery during `setUpTraits()`

Verified against `vendor/laravel/framework/src/Illuminate/Foundation/Testing/
Concerns/InteractsWithTestCaseLifecycle.php:204-242`: Laravel invokes
`setUp<TraitBaseName>` / `tearDown<TraitBaseName>` (the `foreach ($uses as $trait)`
loop), **not** `init<TraitName>`. There is no `init<TraitName>` convention in
Laravel 11.

### Deviation 2 — even `setUp<TraitName>` runs TOO LATE (Rule 1: bug)

Even if we renamed to `setUpSharesPdoAcrossTestConnections`, that hook runs
AFTER RefreshDatabase's `refreshDatabase()` has already opened transactions
on per-connection PDOs. `setUpTraits()` calls `$this->refreshDatabase()` FIRST
(line 209), then loops for `setUp*` methods (line 232).

`beforeRefreshingDatabase()` is callable EARLIER (top of `refreshDatabase()`)
but is **shadowed by PHP trait resolution**: when a child test class uses
`RefreshDatabase` via Pest's `uses()` helper, RefreshDatabase's own (empty)
`beforeRefreshingDatabase()` overrides any inherited method from the parent
TestCase.

**Fix:** Override `setUpTraits()` on `TestCase` itself. Child test classes
never redeclare it, so the trait's override is guaranteed to run. We rebind
PDOs in the override BEFORE calling `parent::setUpTraits()` — which means
rebinding happens before RefreshDatabase's first transaction.

### Deviation 3 — nested BEGINs on shared PDO + search_path drift (Rule 3: blocking)

After rebinding, if `$connectionsToTransact` includes the siblings, Laravel
calls `$sibling->beginTransaction()` on each. PostgreSQL rejects nested BEGINs
on one session — second sibling's BEGIN fails.

The plan's implicit model (reuse of wrapper's grammar, PDO shared) was
correct; the execution mechanics needed two more tweaks:

1. **Prune siblings from `$connectionsToTransact`** in the trait. Since all
   siblings share master's PDO, their writes participate in master's
   transaction naturally — Laravel only needs to BEGIN/ROLLBACK on master.

2. **Apply UNION search_path to the shared PDO** after rebinding. Each
   sibling's connection config declares a search_path (finngen,vocab,php
   vs app,php,public vs inpatient,inpatient_ext,vocab), but a shared PDO
   has ONE session search_path. We set it to the union of all schemas so
   queries from any wrapper resolve to the correct schema by unqualified
   name. Parthenon's schemas are disjoint (app tables ≠ finngen tables
   ≠ inpatient tables) so the union is unambiguous.

## Test-fixture drift auto-fixes (Rule 2)

Two previously-failing tests exposed drift when the FK block was closed.
These were fixed inline per the plan's "shallow test bugs" policy:

### 1. `EndpointBrowserFinngenSchemaTest` — missing Spatie role seed

Previous behavior: FK violation aborted test before `$user->assignRole('super-admin')`.
Now visible: `RoleDoesNotExist: super-admin` because the test ran
`User::factory()->create()` without any role seed.

**Fix:** Added `beforeEach(fn() => $this->seed(RolePermissionSeeder::class))`.

### 2. `IsolationRollbackTest` — write leak to live DEV parthenon

The test uses `DB::connection('finngen')` directly (line 65) to seed the
Phase-13 snapshot. Under pre-13.2-06 conditions, that connection pointed
at the CORRECT parthenon_testing (because Dotenv didn't overwrite the
Docker-set DB_DATABASE). Post-13.2-06, the `finngen` connection correctly
points at DEV parthenon (per the hard-coded `env('DB_DATABASE')` default),
which means this test now writes rows `cohort_definition_id=9000000..9000004`
into LIVE PARTHENON. Subsequent test runs hit unique violations on those
leaked rows.

**Fix:** Route via `config('finngen.connection', 'finngen')` so the test
writes land in `parthenon_testing` under pest (consistent with Plan 13.2-06's
indirection). Production behavior unchanged (fallback to `finngen`).

**Cleanup:** Removed 5 leaked rows from live DEV `parthenon.finngen.endpoint_expressions_pre_phase13`
during debug (`WHERE cohort_definition_id >= 9000000`).

## Column-constraint evidence (architectural gap for follow-on plan)

The 3 EndpointGenerateCohortIdTest failures trace to:

```
$ psql -h localhost -U claude_dev parthenon_testing \
    -c "\d+ finngen.endpoint_generations" | grep cohort_definition_id
 cohort_definition_id  | bigint | not null
```

The EndpointBrowserController post-D-09 inserts `cohort_definition_id = NULL`
(the R worker derives a synthetic id from `generation_id + OMOP_COHORT_ID_OFFSET`
later). But the column still has `NOT NULL`. Plan 13.2-01's migration dropped
NOT NULL on `run_id` only — it did NOT touch `cohort_definition_id`.

**Follow-on action:** New migration `ALTER TABLE finngen.endpoint_generations
ALTER COLUMN cohort_definition_id DROP NOT NULL`. Parallel structure to
`2026_04_20_000100_finngen_endpoint_generations_run_id_nullable.php`.
