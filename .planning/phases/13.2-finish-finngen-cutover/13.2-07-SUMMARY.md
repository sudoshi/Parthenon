### Task 2 — Wire + verify + fix architectural gaps (be75463f9)

Wired the trait into `TestCase.php` via `use` statements (acceptance criteria: exactly 2 grep matches — green). Then ran the full FinnGen Pest suite.

**First run:** 62 failed, 241 passed — SAME AS BASELINE. The trait's `init<TraitName>` method was never called.

**Diagnosis iteration 1 (Deviation 1 — Rule 1 bug):**
Verified against `vendor/laravel/framework/src/Illuminate/Foundation/Testing/Concerns/InteractsWithTestCaseLifecycle.php:204-242`:
Laravel invokes `setUp<TraitBase>` / `tearDown<TraitBase>` via the `foreach ($uses as $trait)` loop (line 232). **There is no `init<TraitName>` convention in Laravel 11.** The plan's central mechanism was based on a framework feature that doesn't exist.

**Diagnosis iteration 2 (Deviation 2 — Rule 1 bug):**
Even renaming to `setUp<TraitName>` doesn't work: that hook runs AFTER `refreshDatabase()` has already opened transactions (line 209 in setUpTraits vs line 232 for the setUp* loop). The correct timing would be `beforeRefreshingDatabase()` (called at the TOP of `refreshDatabase()`) — but PHP's trait resolution shadows that method when the child test class uses `RefreshDatabase` via Pest's `uses()` helper.

**Fix:** Override `setUpTraits()` on `TestCase` itself. Child test classes never redeclare `setUpTraits()`, so the trait's override is guaranteed to run. We rebind PDOs BEFORE `parent::setUpTraits()` triggers RefreshDatabase.

**Diagnosis iteration 3 (Deviation 3 — Rule 3 blocking):**
After rebinding PDOs, RefreshDatabase called `beginTransaction()` on each `$connectionsToTransact` entry. The master (`pgsql_testing`) issued a real BEGIN — good. Each sibling then tried to also issue BEGIN on the same shared PDO — PG rejected with "there is already a transaction in progress".

Initial attempt: use Reflection to bump sibling `Connection::$transactions` to 1 so subsequent `beginTransaction()` took the savepoint branch. This worked for BEGIN but broke on teardown: siblings tried to `ROLLBACK TO SAVEPOINT tans2` after master had already issued `ROLLBACK`, tripping `SQLSTATE[25P01]: No active sql transaction`.

**Better fix:** Prune sibling *_testing names from `$connectionsToTransact` in the trait. RefreshDatabase only opens a transaction on `pgsql_testing` (the master), and sibling writes participate in master's transaction automatically (they share one PG backend session). Master's rollback at teardown rolls back everything. Simpler and robust.

**Diagnosis iteration 4 (search_path drift):**
After the transaction mechanics worked, queries started failing with
`relation "analysis_modules" does not exist`. The sibling's query ran through
the master's PDO, which had the master's `app,php,public` search_path —
not the sibling's `finngen,vocab,php` search_path.

**Fix:** Apply a UNION search_path to the shared PDO at rebind time:
`SET SESSION search_path TO app,php,public,finngen,vocab,inpatient,inpatient_ext`.
This works because Parthenon's schemas are DISJOINT (no table name collisions
between app/finngen/inpatient/vocab/php), so the union is unambiguous and
every query resolves to the correct schema by unqualified name.

**Final run:** 62 → 6 failures, 241 → 297 passes. The 60-test FK-visibility gap is closed.

## Deviations from Plan

### Rule 1 bug — `init<TraitName>` does not exist in Laravel 11

- **Found during:** Task 2 first Pest run (62 failures, no change from baseline).
- **Issue:** Plan's central mechanism relied on a non-existent Laravel convention.
- **Fix:** Overrode `setUpTraits()` on TestCase instead (guaranteed-to-run hook).
- **Files modified:** `backend/tests/Concerns/SharesPdoAcrossTestConnections.php`
- **Commit:** `be75463f9`

### Rule 1 bug — `setUp<TraitName>` hook runs too late

- **Found during:** Iterating on the Rule 1 fix above.
- **Issue:** The correct convention (`setUp<TraitBase>`) fires AFTER RefreshDatabase's transaction opens — rebinding at that point corrupts the active transaction.
- **Fix:** Same as above — `setUpTraits()` override.
- **Commit:** `be75463f9`

### Rule 3 blocking — nested BEGINs + search_path drift

- **Found during:** Task 2 iteration (post-Rule-1 fix).
- **Issue:** Siblings sharing PDO triggered PG's "nested BEGIN" rejection; shared PDO's single search_path caused `relation does not exist` on sibling queries.
- **Fix:** Prune siblings from `$connectionsToTransact` + UNION search_path.
- **Files modified:** same trait file.
- **Commit:** `be75463f9`

### Rule 2 auto-fix — missing Spatie role seed in EndpointBrowserFinngenSchemaTest

- **Found during:** Task 2 full-suite run (after trait was working).
- **Issue:** `$user->assignRole('super-admin')` threw `RoleDoesNotExist` — previously masked by the FK violation aborting tests earlier.
- **Fix:** Added `beforeEach(fn() => $this->seed(RolePermissionSeeder::class))`.
- **Files modified:** `backend/tests/Feature/FinnGen/EndpointBrowserFinngenSchemaTest.php`
- **Commit:** `be75463f9`

### Rule 2 auto-fix — IsolationRollbackTest write-leak to live DEV parthenon

- **Found during:** Task 2 full-suite run.
- **Issue:** Test used `DB::connection('finngen')` directly (hardcoded production connection). Under Plan 13.2-06's post-environment setup, this writes to LIVE parthenon, not parthenon_testing. Previous test runs leaked rows (`cohort_definition_id=9000000..9000004`) into live DB, causing unique violations on subsequent runs.
- **Fix:** Route via `config('finngen.connection', 'finngen')` for test-aware indirection (same pattern as Plan 13.2-06 models + controllers).
- **Cleanup:** Removed 5 leaked rows from live `parthenon.finngen.endpoint_expressions_pre_phase13`.
- **Files modified:** `backend/tests/Feature/FinnGen/IsolationRollbackTest.php`
- **Commit:** `be75463f9`
