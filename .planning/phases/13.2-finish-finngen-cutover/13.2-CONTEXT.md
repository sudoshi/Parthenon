<decisions>
## Implementation Decisions

### Cohort-table ID strategy (core architectural question)
- **D-01:** Use `finngen.endpoint_generations.id + 100_000_000_000` as the OMOP `{cohort_schema}.cohort.cohort_definition_id` for FinnGen-generated cohorts. Rationale: (a) already tracked per (endpoint_name, source_key) pair; (b) `finngen.endpoint_generations.id` is a bigserial and will never overflow `bigint` at any realistic scale (trillions of generations would still fit); (c) offset of 1e11 puts FinnGen keys well above any realistic `app.cohort_definitions.id` high-water mark (current DB has ~122 user cohorts; 1B would be safe, 100B is absurdly safe); (d) deterministic — the same generation row always maps to the same cohort_definition_id, which lets the existing overwrite_existing gate work correctly.
- **D-02:** The offset constant is a single app-level definition: `App\Models\App\FinnGenEndpointGeneration::OMOP_COHORT_ID_OFFSET = 100_000_000_000` (int const, not config). Reused in: the R worker (hard-coded as a matching constant with a comment pointing at the PHP def), the Laravel controller when passing params, and any Pest tests that assert on the written cohort_definition_id. Single source of truth, no env drift.
- **D-03:** R worker accepts both keys for backward compat: if `params$finngen_endpoint_generation_id` is present AND > 0, compute `cohort_def_id = finngen_endpoint_generation_id + 100000000000L`. Otherwise fall back to `params$cohort_definition_id` as before. Rejects the run ONLY if BOTH are null/invalid. This keeps the legacy path intact for non-FinnGen Darkstar analysis types (cohort.materialize, etc.).
- **D-04:** Controller wiring: after `FinnGenEndpointGeneration::updateOrCreate(...)` returns, refetch with `->fresh()` to get the PK, then include `finngen_endpoint_generation_id => $generation->id` in the params dict passed to `FinnGenRunService::create()`. Keep `cohort_definition_id => null` in the same dict for transparency.

### Role-split grants codification
- **D-05:** Create a regular migration (not a `--path=` hotfix) at `backend/database/migrations/2026_04_20_NNNNNN_codify_phase_13_1_role_split_baseline.php`. It runs automatically on any fresh environment (parthenon_testing included) via the normal `php artisan migrate` sequence. Idempotent via `DO $$ ... IF ... THEN ... END IF; $$` guards that check current state before applying.
- **D-06:** The migration's 3 statements:
  1. `ALTER TABLE app.cohort_definitions OWNER TO parthenon_migrator` — guarded by `(SELECT tableowner FROM pg_tables WHERE schemaname='app' AND tablename='cohort_definitions') != 'parthenon_migrator'`
  2. `GRANT SELECT,INSERT,UPDATE,DELETE,TRUNCATE,REFERENCES,TRIGGER ON app.cohort_definitions TO parthenon_app` — always safe to re-grant (idempotent)
  3. `GRANT CREATE ON DATABASE parthenon TO parthenon_migrator` — use `current_database()` so the migration is portable to parthenon_testing (`GRANT CREATE ON DATABASE <SCHEMA_NAME> TO parthenon_migrator` via dynamic SQL)
- **D-07:** The migration runs as `parthenon_migrator` (per deploy.sh pattern). This works for grants because `parthenon_migrator` is a member of `parthenon_owner` which owns `app.cohort_definitions`. For the `GRANT CREATE ON DATABASE`, migrator may need superuser — if so, document the requirement in the migration's docstring and in the deploy runbook.

### Test coverage strategy
- **D-08:** Full `parthenon_testing` Pest sweep is non-negotiable — running `docker compose exec -T php vendor/bin/pest tests/Feature/FinnGen tests/Unit/FinnGen --no-coverage` must exit 0. Any pre-existing failures (the 4 known Mockery type errors in `ImportEndpointsCommandTest` from 13.1-04) either get fixed or get marked with `->todo()` / `@skip` with a clear docstring pointing at a deferred-items issue. Not blocking.
- **D-09:** A new Pest test `tests/Feature/FinnGen/EndpointGenerateCohortIdTest.php` asserts the controller's output: given a fresh EndpointDefinition + Source, hit the generate endpoint, verify the returned run params contain `finngen_endpoint_generation_id > 0` and that `cohort_definition_id` key is `null`. Keeps the collision-free-offset contract under CI protection.
- **D-10:** The PANCREAS smoke-gen E2E verification is a manual/curl check captured in a DEPLOY-LOG addendum (not a Pest test — it requires a live Darkstar worker + PANCREAS data source + admin token). Documented with run_id, status transitions, final subject_count, and the actual cohort_definition_id that landed in `pancreas.cohort`.
- **D-11:** Spot-check regression: generate a non-FinnGen user cohort (standard `app.cohort_definitions`-backed flow) and confirm it still writes to `{source}.cohort` with its real `app.cohort_definitions.id`. Prevents the R worker change from silently breaking the legacy path.

## Existing Code Insights

### Reusable Assets
- **`FinnGenEndpointGeneration` model** (existing, Phase 13.1-03-wired) — has bigserial PK `id`, connection `finngen`, table `endpoint_generations`. The `id` column is the ideal OMOP key source per D-01.
- **`FinnGenRunService::create()`** — already accepts arbitrary params dict; passing a new `finngen_endpoint_generation_id` field is additive.
- **Darkstar R `run_with_classification()` wrapper** — existing error-classification + progress-reporting envelope. 13.2 R worker edit reuses this unchanged.
- **`DatabaseConnector::executeSql/querySql`** — existing R DB driver. Unchanged — only the cohort_def_id value differs.
- **Idempotent migration pattern** — Parthenon's HIGHSEC migrations consistently use `DO $$ BEGIN IF <guard> THEN <ddl>; END IF; END $$;` blocks. 13.2 grants migration follows the pattern.

### Established Patterns
- **Constant as int literal (D-02):** PHP uses `100_000_000_000` (integer literal with underscores); R uses `100000000000L` (integer suffix). Both fit in 64-bit signed int.
- **`updateOrCreate(...)->fresh()`** — Laravel idiom for getting the persisted row with auto-assigned PK. Used elsewhere in Parthenon (check CohortGenerationService for reference).
- **Pest `RefreshDatabase` trait** — runs all migrations on parthenon_testing at test boot. The 13.1 isolate_finngen_schema migration + the 13.2 role-grants migration both run here, proving SC 10.

### Integration Points
- **`EndpointBrowserController::generate()`** — single point of wiring for the new generation_id param.
- **Darkstar `finngen.endpoint.generate` analysis type dispatcher** — the params dict flows from Laravel → Horizon job → Darkstar R worker unchanged. Only the R worker's param reading changes.
- **`{pancreas,omop,synpuf,...}.cohort` tables** — receive rows with the new offset-keyed cohort_definition_id. No DDL change; just a new numeric range of ids.
- **Atlas / Analyses / Abby queries** — ANY query that joins `{source}.cohort.cohort_definition_id` to `app.cohort_definitions.id` will NOT find FinnGen-generated rows (by design — FinnGen cohorts are not in `app.cohort_definitions`). This is a feature, not a bug. D-11 spot-check confirms the inverse direction (user cohorts still link).

<specifics>
## Specific Ideas

- **100_000_000_000 offset is deliberately excessive** — current DB has 122 user cohorts; even projecting to 10M cohorts over a decade leaves 5 orders of magnitude of headroom before colliding with the FinnGen range. Pragmatic safety margin.
- **The R worker change is ADDITIVE not a full refactor** — preserves the legacy `cohort_definition_id`-backed code path for non-FinnGen Darkstar analysis types (cohort.materialize, CohortMethod, etc.). Only the FinnGen `endpoint.generate` analysis type uses the new key.
- **The role-grants migration is a pure codification exercise** — the 3 statements were already applied to dev manually (captured in 13.1-DEPLOY-LOG.md); this phase just writes them into a migration file so fresh environments get the same state.
- **`parthenon_testing` full Pest sweep is the SC 10 gate** — Phase 13.1 deferred SC 10 to "tests-green milestone"; this phase IS that milestone.
- **E2E smoke-gen is the final proof** — until an actual `status=succeeded` with `subject_count > 0` lands in `pancreas.cohort`, 13.1's Phase 13 invariant preservation is unproven. 13.2 proves it.

</specifics>
