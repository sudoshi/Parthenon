# Phase 13.2 Plan 04: Wave 3 Validation Gate — PARTIAL (MIGRATION GREEN, SUITE RED)

Fixed the migration idempotency regression flagged by Plans 01/02/03
(`SQLSTATE[42P07]` at `isolate_finngen_schema.php:65`). Also fixed a
fixture drift in `ImportEndpointsCommandTest`. After the fixes:
`migrate:fresh` on `parthenon_testing` succeeds cleanly; SC-10 from
Phase 13.1 is satisfied at the schema level
(`FinnGenSchemaExistsInTestDbTest` and `IsolationMigrationTest` pass).

However, 71 tests (48 Feature + 23 Unit) remain failing due to two
pre-existing architectural gaps — neither of which is within this plan's
explicit `<parallel_execution>` scope:

1. **Cross-connection transaction routing** — the `finngen` connection
   targets live DEV `parthenon` (not `parthenon_testing`) under
   `APP_ENV=testing`, and the models hardcode `$connection = 'finngen'`.
   Needs a `finngen_testing` connection + 6 model overrides.
2. **Missing `GRANT USAGE ON SCHEMA app TO parthenon_migrator`** on
   `parthenon_testing`. Codified in DEV but not in Plan 13.2-02's migration.

Both fixes involve files explicitly forbidden by this plan's scope
(Wave 1/2/3 models + 13.2 migrations). Returning the plan with a
structured checkpoint so the orchestrator can route to a follow-on plan.

### Rule 3 auto-fix — stub drift in ImportEndpointsCommandTest (Task 1)

**Found during:** Task 1 verification (initial Pest run after migration fix).

**Issue:** `ImportEndpointsCommandTest::finngenStubResolver()` mocked only
4 of 7 resolver methods. `FinnGenEndpointImporter::processEndpoint()` calls
`resolveIcdO3` at line 205, `resolveNomesco` at 206, `resolveKelaReimb` at
207 — Mockery throws `BadMethodCallException` on strict-mode calls with no
expectation.

**Fix:** Added 3 `shouldReceive(...)->andReturn(...)` entries mirroring the
existing `resolveIcd8` stub pattern (empty-result + `truncated: false`).
Same `@phpstan-ignore method.notFound` treatment as 13.2-01's precedent.
Fixture drift — fits plan scope ("test fixture file ONLY if a test failure
traces to fixture drift"). Recovered 6 test cases from BadMethodCallException.

**Files modified:** `backend/tests/Feature/FinnGen/ImportEndpointsCommandTest.php`
**Commit:** `5ef5a7aef` (same commit as migration fix).

### Analytical stop — Rule 4 architectural escalation (71 remaining failures)

**Found during:** Post-fix Pest re-run.

**Finding:** 41 FK violations on `finngen_runs_user_id_fkey`, 3 unique
constraint violations on `endpoint_definitions_pkey`, 8 cascading HTTP
500s, 1 permission-denied on `app` schema, and ~18 other failures
(assertion mismatches stemming from the same root cause).

**Root cause:** Two pre-existing architectural gaps (detailed in PEST-LOG
§Architectural Blocker + §Secondary Blocker):
1. The `finngen` connection is hardcoded to `env('DB_DATABASE', 'parthenon')`
   and points at the live DEV DB even in testing — cross-connection
   isolation breaks every FinnGen model that writes through the `finngen`
   connection.
2. Missing `GRANT USAGE ON SCHEMA app TO parthenon_migrator` on
   parthenon_testing (grant exists in DEV, not codified in Plan 02's
   migration).

**Why NOT auto-fixed:** Both fixes require modifications to Wave 1/2/3
files (FinnGen models + Plan 02 migration) which this plan's
`<parallel_execution>` section explicitly FORBIDS:
> Do NOT modify the 13.2 migrations (Plans 01 + 02). Do NOT modify any
> code from Waves 1/2/3 (controller, R worker, models).

**Escalation path:** Opened a follow-on-plan recommendation in the
PEST-LOG's Deferred Issues section. The orchestrator should schedule a
13.2-06 (or fold into 13.3) to:
- Add `finngen_testing` connection in `backend/config/database.php`
  (mirrors `inpatient_testing` pattern established by Morpheus).
- Update 6 FinnGen model `$connection` fields to read from config.
- Add a follow-on role-split-completion migration with
  `GRANT USAGE ON SCHEMA app TO parthenon_migrator`.
- Add `$connectionsToTransact = ['pgsql_testing', 'finngen_testing']`
  to tests/TestCase.php (or new FinnGenTestCase base class).

**Plan 05 (Wave 4) impact:** None. Plan 05's PANCREAS smoke-gen runs
against DEV `parthenon` where the finngen connection + the role grants
are already correctly configured. Plan 05 does not depend on
parthenon_testing being fully green.
