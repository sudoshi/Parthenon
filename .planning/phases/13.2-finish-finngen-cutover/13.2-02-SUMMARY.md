# Phase 13.2 Plan 02: Codify Role-Split Baseline — Summary

## One-Liner

Codified the 3 Phase 13.1 dev-applied role-split deviations (D4 + D5 + D6 from `13.1-DEPLOY-LOG.md`) into a single idempotent Laravel migration. Fresh environments (parthenon_testing bootstrap, future staging/prod) now converge on the same `parthenon_migrator`-owned `app.cohort_definitions` state with explicit `parthenon_app` DML grants and a best-effort `GRANT CREATE ON DATABASE`.

## What Was Built

**1 new migration file** at `backend/database/migrations/2026_04_20_000200_codify_phase_13_1_role_split_baseline.php` containing 3 DDL statements:

### Statement 1 (D5) — guarded ownership transfer
```sql
DO $mig$
BEGIN
  IF (SELECT tableowner FROM pg_tables
        WHERE schemaname = 'app' AND tablename = 'cohort_definitions'
     ) IS DISTINCT FROM 'parthenon_migrator' THEN
    ALTER TABLE app.cohort_definitions OWNER TO parthenon_migrator;
  END IF;
END;
$mig$;
```
Idempotent via the `IS DISTINCT FROM` pre-check. No-op on DEV (owner is already `parthenon_migrator` after the 13.1 manual cutover).

### Statement 2 (D6) — re-grant DML to runtime role
```sql
GRANT SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER
  ON app.cohort_definitions TO parthenon_app;
```
Naturally idempotent — PG treats re-grants as no-ops. Restores runtime DML surface after the ownership transfer in Statement 1 drops the prior owner's implicit rights.

### Statement 3 (D4) — EXCEPTION-wrapped `GRANT CREATE ON DATABASE`
```sql
DO $mig$
BEGIN
  EXECUTE format('GRANT CREATE ON DATABASE %I TO parthenon_migrator', current_database());
EXCEPTION WHEN insufficient_privilege THEN
  RAISE NOTICE 'Phase 13.2: migration role cannot self-grant CREATE ON DATABASE %; apply manually as DB owner or superuser.', current_database();
END;
$mig$;
```
Uses `EXECUTE format(..., current_database())` because PG requires a literal identifier for `GRANT ... ON DATABASE <name>`. The `EXCEPTION WHEN insufficient_privilege` handler ensures the migration runs cleanly on `parthenon_testing` (where the migration role is not the DB owner), emitting a NOTICE that the grant must be applied manually — documented in the 13.2 deploy runbook (produced in Plan 05).

`down()` is a deliberate no-op with a docblock pointer to the runbook: these are additive hardening grants; reversing them would break runtime.

### 1. [Rule 1 - Planner oversight] grep-count criterion for `GRANT CREATE ON DATABASE` is `= 1` but prescribed file body produces `= 3`

**Found during:** Task 1 acceptance-criteria verification
**Issue:** The plan's `<acceptance_criteria>` block and the orchestrator's `<success_criteria>` both state:
`grep -c 'GRANT CREATE ON DATABASE' ... returns 1`
However, the plan's `<action>` block prescribes the exact file content verbatim — that content contains 3 matches for the string `GRANT CREATE ON DATABASE` (docblock at line 20, code comment at line 56, EXECUTE statement at line 66).
**Resolution:** The prescribed file content is the authoritative specification (it matches RESEARCH §Example 3 verbatim and reflects the intent to document the operation in the docblock, comment, and the actual SQL). Interpreted the grep-count criterion as "at least 1" — consistent with the companion criterion `grep -c 'current_database()' ... returns at least 2`, which uses the same "at least" phrasing for the same comment/SQL overlap pattern.
**Files modified:** None (no code change required)

**Found during:** Task 1 Pint/PHPStan verification

**Resolution:** Used `docker cp` to copy the migration into the container as `/tmp/codify.php`, ran `vendor/bin/pint --test` on that path, then briefly placed it at the expected path to run `phpstan analyse` and `migrate:status` (phpstan resolves from the real filesystem path).
