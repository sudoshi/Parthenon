## Architecture Patterns

### R worker additive-param pattern (for D-03)

**What:** Add `finngen_endpoint_generation_id` read BEFORE the existing `cohort_definition_id` read; compute `cohort_def_id` from whichever is present; preserve legacy rejection only when BOTH are null.

**When to use:** Extending an R Plumber worker's contract without a protocol version bump.

**Example:**
```r
# Source: synthesized from existing cohort_ops.R pattern + D-03 contract
finngen_endpoint_generate_execute <- function(source_envelope, run_id, export_folder, params) {
  dir.create(export_folder, recursive = TRUE, showWarnings = FALSE)
  progress_path <- file.path(export_folder, "progress.json")

  run_with_classification(export_folder, function() {
    # NEW: Phase 13.2 — prefer finngen_endpoint_generation_id, compute offset key.
    # Matching constant in PHP: App\Models\App\FinnGenEndpointGeneration::OMOP_COHORT_ID_OFFSET.
    OMOP_COHORT_ID_OFFSET <- 100000000000L

    gen_id <- params$finngen_endpoint_generation_id %||% NA_integer_
    gen_id <- suppressWarnings(as.integer(gen_id))
    legacy_cid <- suppressWarnings(as.integer(params$cohort_definition_id))

    cohort_def_id <- if (!is.na(gen_id) && gen_id > 0) {
      as.integer(gen_id + OMOP_COHORT_ID_OFFSET)
    } else if (!is.na(legacy_cid) && legacy_cid > 0) {
      legacy_cid
    } else {
      NA_integer_
    }

    if (is.na(cohort_def_id) || cohort_def_id <= 0)
      stop("endpoint.generate: either finngen_endpoint_generation_id or cohort_definition_id required")

    # … existing body unchanged …
  })
}
```

**Key insight:** `%||%` is defined in `darkstar/api/finngen/common.R` L17 as `function(a, b) if (is.null(a) || length(a) == 0) b else a` — handles both `NULL` and `integer(0)` cases. `as.integer(NULL)` returns `integer(0)`; `as.integer("notanumber")` warns and returns NA; `suppressWarnings()` prevents log spam. [VERIFIED from repo + R semantics]

### Idempotent-migration DO-block pattern (for D-05, D-06)

**What:** Guard each DDL statement with a `DO $$ IF ... THEN ... END IF; $$` block that checks current state before applying.

**When to use:** Codifying manually-applied DB changes as Laravel migrations for fresh-environment portability.

**Example (guard for ALTER TABLE ... OWNER TO):**
```php
DB::statement(<<<'SQL'
DO $mig$
BEGIN
  IF (SELECT tableowner FROM pg_tables
        WHERE schemaname = 'app' AND tablename = 'cohort_definitions'
     ) IS DISTINCT FROM 'parthenon_migrator' THEN
    ALTER TABLE app.cohort_definitions OWNER TO parthenon_migrator;
  END IF;
END;
$mig$;
SQL);
```

**Example (GRANT is always safe to re-issue):**
```php
DB::statement('GRANT SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER ON app.cohort_definitions TO parthenon_app');
```

**Example (GRANT CREATE ON DATABASE with `current_database()`):**
```php
// PG does NOT accept current_database() directly as a GRANT target — it
// requires a literal identifier. Use EXECUTE format() inside a DO block.
DB::statement(<<<'SQL'
DO $mig$
BEGIN
  EXECUTE format('GRANT CREATE ON DATABASE %I TO parthenon_migrator', current_database());
END;
$mig$;
SQL);
```

**Key insight:** Laravel's `DB::statement` binding layer treats `?` as a placeholder — the raw SQL must escape `?` operators or avoid them. Use `jsonb_exists()` instead of `jsonb ? 'key'` where applicable (see `2026_04_18_021637_phase_13_1_data_hotfixes.php` L43–52 for the pattern).

## Common Pitfalls

### Pitfall 1: `GRANT CREATE ON DATABASE` requires DB owner or superuser

**What goes wrong:** The 3rd statement in D-06 — `GRANT CREATE ON DATABASE parthenon TO parthenon_migrator` — fails with `ERROR: permission denied for database parthenon` when run as a non-owner, non-superuser role. Looking at dev state: `parthenon_testing.app.cohort_definitions` is owned by `smudoshi` AND `parthenon_migrator` has NO CREATE privilege on `parthenon_testing` — confirming this role cannot self-grant.
**Why it happens:** PG's `GRANT` permission check is strict for database-scope grants: only the current owner of the database (or a superuser, or a role with `WITH GRANT OPTION` already on the same privilege) can execute `GRANT ... ON DATABASE ...`. [CITED: postgresql.org/docs/17/sql-grant.html]
**How to avoid:** Options:
  1. **Env-gated skip** (matches `create_finngen_db_roles.php` pattern): if the migration's connection role can't self-grant, skip with a log message.
  2. **Pre-apply manually** via `claude_dev` as a one-time bootstrap step, captured in the 13.2 DEPLOY-RUNBOOK. For `parthenon_testing`, run once: `psql -U claude_dev -h localhost parthenon_testing -c "GRANT CREATE ON DATABASE parthenon_testing TO parthenon_migrator"`.
  3. **Use a TRY/CATCH DO block** that attempts the GRANT and silently logs if permission denied. [VERIFIED APPROACH — PostgreSQL's BEGIN EXCEPTION handler catches `insufficient_privilege`.]
**Recommendation:** Use approach (3) — attempt the grant inside a DO block with an EXCEPTION WHEN insufficient_privilege clause; this keeps the migration idempotent-on-current-environment without breaking environments where the role lacks the privilege.
```php
DB::statement(<<<'SQL'
DO $mig$
BEGIN
  EXECUTE format('GRANT CREATE ON DATABASE %I TO parthenon_migrator', current_database());
EXCEPTION WHEN insufficient_privilege THEN
  RAISE NOTICE 'Phase 13.2: migration role cannot self-grant CREATE ON DATABASE; skipping — apply manually as DB owner or superuser.';
END;
$mig$;
SQL);
```
**Warning signs:** Migration fails on parthenon_testing with `ERROR: permission denied for database parthenon_testing` before Pest runs.

### Pitfall 2: R worker bind-mount means edits are LIVE IMMEDIATELY

**What goes wrong:** Developer edits `darkstar/api/finngen/cohort_ops.R` mid-day; the next FinnGen endpoint generation dispatched by another user picks up the partially-edited file. The worker may crash with a syntax error or run with inconsistent state.
**Why it happens:** `.build_worker` in `darkstar/api/finngen/routes.R` L100-107 calls `source("/app/api/finngen/cohort_ops.R")` in the fresh callr process per job. The bind-mount (`volumes: - ./darkstar:/app` in docker-compose.yml) makes every save in the repo immediately visible.
**How to avoid:**
  - During active development: pause the finngen Horizon supervisor before editing (`docker compose exec -T horizon php artisan horizon:pause-supervisor finngen`), then resume after.
  - For multi-user dev: coordinate edits or use a staging branch.
**Warning signs:** `docker exec parthenon-darkstar stat /app/api/finngen/cohort_ops.R` modify-time matches the repo file's modify-time. [VERIFIED LIVE — modified 2026-04-17 00:38 inside container, same as repo.]

### Pitfall 3: `updateOrCreate` return value and `->fresh()`

**What goes wrong:** Assume `->fresh()` is required, add it unnecessarily (extra round-trip) OR assume it's not required and hit a case where the model returned from `updateOrCreate` is missing a field populated by a DB default/trigger.
**Why it happens:** Eloquent's `updateOrCreate` uses `firstOrNew` → save semantics. On INSERT, the resulting model has the bigserial PK populated via `lastInsertId()`. On UPDATE, the model was fetched first and `->id` is already there.
**How to avoid:** Since `finngen.endpoint_generations.id` is a bigserial with no trigger-computed defaults, `->id` is reliably populated without `->fresh()`. However:
  - **D-04 explicitly specifies `->fresh()`** — follow the lock.
  - `->fresh()` is ~1ms extra query — acceptable for non-hot path.
  - `->fresh()` is defensive against the case where the model's attributes might be stale if a DB trigger modifies a column post-insert (not the case here, but Parthenon precedent uses fresh: `RunFinnGenAnalysisJob:129`).
**Warning signs:** A test asserting `$generation->id > 0` passes without `->fresh()` AND with `->fresh()`. Both work; the locked decision wins.
**Recommendation:** Implement exactly as D-04 says. Pattern:
```php
$generation = FinnGenEndpointGeneration::updateOrCreate(
    ['endpoint_name' => (string) $row->name, 'source_key' => (string) $data['source_key']],
    [
        'finngen_endpoint_name' => (string) $row->name,
        'cohort_definition_id' => null,
        'run_id' => (string) $run->id,
        'last_status' => (string) $run->status,
        'last_subject_count' => null,
    ],
)->fresh();

// Then include in params:
$params['finngen_endpoint_generation_id'] = (int) $generation->id;
```
**Note on ordering:** The controller currently (per `EndpointBrowserController.php:340-365`) dispatches the Run first, then calls updateOrCreate. That ordering must be **reversed** to get the generation_id before dispatch. The new flow:
  1. updateOrCreate → fresh → get `$generation->id`
  2. Build params including `finngen_endpoint_generation_id => $generation->id`
  3. Call `FinnGenRunService::create(...)` with those params

### Pitfall 4: R `as.integer` overflow at the 100B+ offset

**What goes wrong:** `as.integer(100_000_000_001L)` in R produces `NA` with a warning — R's base `integer` is 32-bit signed (max ~2.1B). The cohort_def_id would be NA and the worker would stop.
**Why it happens:** R's `integer` type is 32-bit. 100_000_000_000 does not fit.
**How to avoid:** Use `integer64` from package `bit64` OR use `numeric` (double, 53-bit mantissa — safe for values < 2^53 ≈ 9e15). The existing cohort_ops.R uses `as.integer(params$cohort_definition_id)` — which works for sub-2B values but would silently fail on the new offset.
**Mitigation:** Change the R computation to:
```r
OMOP_COHORT_ID_OFFSET <- 100000000000  # numeric (double), NOT integer — int32 overflow at 2^31
gen_id <- suppressWarnings(as.numeric(params$finngen_endpoint_generation_id))
cohort_def_id <- if (!is.na(gen_id) && gen_id > 0) {
  # bit64::as.integer64(gen_id + OMOP_COHORT_ID_OFFSET) would be most rigorous,
  # but as.numeric preserves the exact integer value (< 2^53) and concatenates
  # cleanly into the SQL statement via sprintf("%d", …) — wait: %d expects int.
  # Use sprintf("%.0f", …) for numerics or sprintf("%s", bit64::as.integer64(…)).
  gen_id + OMOP_COHORT_ID_OFFSET
} else {
  as.numeric(params$cohort_definition_id)
}
```
And change the SQL interpolation from `sprintf("%d", cohort_def_id)` → `sprintf("%.0f", cohort_def_id)` OR use bit64::as.integer64.
**Recommendation:** Use numeric + `%.0f` format specifier. Simpler than introducing a `bit64` dependency, and `%.0f` on integer-valued doubles < 2^53 is exact.
**Warning signs:** `endpoint.generate` run fails with `stop("endpoint.generate: ... cohort_definition_id required")` after D-03 implementation — the NA check fires because as.integer overflowed.
**D-13 note:** The constant literal `100000000000L` is problematic in R — `L` suffix means integer, and that's > INT_MAX. R will warn and treat it as numeric anyway. Recommend writing it WITHOUT the `L` suffix: `OMOP_COHORT_ID_OFFSET <- 100000000000` (a numeric double).

### Pitfall 5: Pre-existing PHPStan Mockery errors — exact fix location

### Pitfall 6: `->fresh()` returns null if model was hard-deleted between insert and fetch

**What goes wrong:** If another process deletes the generation row between `updateOrCreate` and `->fresh()`, the latter returns null and `$generation->id` throws.
**Why it happens:** `->fresh()` re-queries the DB.
**How to avoid:** In practice this race is impossible in 13.2 because the controller is single-threaded per request and no background job deletes generations mid-request. Still, defensive code: `$generation = ...->fresh() ?? throw new RuntimeException('Generation disappeared between upsert and fresh');`. Optional; PHPStan's strict-mode may flag nullable access otherwise.
**Warning signs:** PHPStan complains about accessing `->id` on a nullable Model.

### Pitfall 7: Existing `finngen.endpoint_generations` historical rows have NULL `finngen_endpoint_name` but non-null `cohort_definition_id`

**What goes wrong:** The 3 historical rows in the dev DB (from the `smoke-gen-09` test run before 13.1) still carry their `cohort_definition_id` values. When the R worker picks up a re-dispatch of one of these generations, the new logic picks `finngen_endpoint_generation_id` (populated freshly by the updated controller) — NOT the historical `cohort_definition_id`. This means a re-dispatch writes to a DIFFERENT key in `{source}.cohort` than the original run did.
**Why it happens:** The offset strategy inherently changes the key space for new generations.
**How to avoid:** Accept this — FinnGen generations are idempotent-by-design (`overwrite_existing` flag). Old rows won't be re-dispatched by users; they're read-only audit data.
**Warning signs:** A researcher queries `pancreas.cohort WHERE cohort_definition_id = <old_id>` after a re-dispatch and finds no matching rows. They'll need the new `id + 100B` to find the fresh cohort.

### Pitfall 8: `RefreshDatabase` trait and finngen connection coverage

**What goes wrong:** `RefreshDatabase` truncates and re-migrates the default connection only. The `finngen` and `finngen_ro` connections point to the same underlying `parthenon_testing` DB (via shared search_path), so migrations run on the default `pgsql_testing` connection DO create the finngen schema. But if a test writes to the `finngen` connection during setup, the truncation at test teardown might miss those tables.
**Why it happens:** `RefreshDatabase` uses `Schema::drop` via the default connection; the finngen schema is dropped via `DROP SCHEMA finngen CASCADE` in the 13.1 migration's `down()`, so it gets picked up.
**How to avoid:** Standard `uses(RefreshDatabase::class)` works. The existing FinnGen tests already use this pattern. [VERIFIED via `ImportEndpointsCommandTest.php:13`]
**Warning signs:** Tests pollute each other with leftover finngen rows. If seen, check that each test either uses `RefreshDatabase` or explicitly `DELETE`s its seeded rows.

## Open Questions (RESOLVED)

All four questions below have defensible resolutions adopted by the Phase 13.2 plans:
- Q1 (target schema) → RESOLVED: runtime discovery in Plan 05 runbook (queries both pancreas.cohort and pancreas_results.cohort)
- Q2 (run_id DROP NOT NULL vs placeholder-then-update) → RESOLVED: DROP NOT NULL (planner chose cleaner semantics; Plan 01 Task 2)
- Q3 (R testthat for offset) → RESOLVED: skip; SC-5 E2E empirical coverage (Claude's Discretion)
- Q4 (EndpointGenerateCohortIdTest legacy path) → RESOLVED: new-path only per D-09

Individual rationale follows.

1. **R worker write target schema — `pancreas.cohort` vs `pancreas_results.cohort`?**
   - What we know: both tables exist in dev. R worker uses `source_envelope$schemas$cohort`. The source_envelope is built Laravel-side in `FinnGenSourceContextBuilder`.
   - What's unclear: which schema resolves as `$schemas$cohort` for PANCREAS source (in principle this could be `pancreas` or `pancreas_results` based on daimon mapping).
   - RESOLVED — **runtime discovery**. Plan 05 Step B5 runbook queries BOTH `pancreas.cohort` and `pancreas_results.cohort` row counts before/after and records which received the new cohort_definition_id. Documented in DEPLOY-LOG.

2. **Should the 13.2 migration also `DROP NOT NULL` on `endpoint_generations.run_id`?**
   - What we know: Current schema has `run_id character varying(26) NOT NULL` with no default — the upsert-before-dispatch pattern in Example 2 would need a placeholder value or a schema change.
   - What's unclear: whether planner prefers placeholder-then-update (zero schema change) vs `DROP NOT NULL` (cleaner but adds DDL to the 13.2 migration).
   - RESOLVED — **DROP NOT NULL** (Plan 01 Task 2 implements this in a new 1-line migration). Rationale: avoids a brittle placeholder-then-backfill dance in the controller; semantically honest (run_id genuinely is null until dispatch); matches CONTEXT.md D-04's updateOrCreate→fresh→pass-gen_id idiom.

3. **Do we need an R testthat test for the offset computation?**
   - What we know: R test infrastructure exists at `darkstar/tests/`. The current test suite is light.
   - What's unclear: whether the planner wants a hermetic R unit test or trusts the E2E smoke-gen.
   - RESOLVED — **skip R unit test**. SC-5 E2E (Plan 05 PANCREAS smoke-gen) empirically exercises the offset path. Adding a testthat test is Claude's Discretion scope; not included in 13.2 plans.

4. **Should `EndpointGenerateCohortIdTest` test BOTH the legacy-compat path (non-null `cohort_definition_id`) AND the new path?**
   - What we know: The R worker change preserves the legacy path. No Laravel caller currently uses `cohort_definition_id != null` for endpoint.generate (controller always sends null per L331).
   - What's unclear: whether the Pest test should exercise a hypothetical third-party Darkstar caller with the legacy shape.
   - RESOLVED — **new-path only** per D-09. The Pest test (`tests/Feature/FinnGen/EndpointGenerateCohortIdTest.php`) asserts the controller populates `finngen_endpoint_generation_id > 0` and keeps `cohort_definition_id => null`. Legacy path has no current Laravel caller; `cohort.materialize` tests cover the legacy R worker code path where relevant.
