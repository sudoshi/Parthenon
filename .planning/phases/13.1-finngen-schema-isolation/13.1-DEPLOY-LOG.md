## Step 2: Pause Horizon finngen supervisor

**Deviation (Rule 3 — blocking issue, auto-fixed):** `docker compose exec -T php php artisan horizon:pause-supervisor finngen` fails with `ERROR: Failed to find a supervisor with this name.` Root cause: `horizon:pause-supervisor` filters supervisors where `Str::startsWith($supervisor->name, MasterSupervisor::basename())`. Inside the `php` container, `basename()` returns the `php` container's hostname (`05fb08a43d57`) — but the running supervisors were spawned by the `horizon` container (`92bf35037e6e-Mw2h`), so the `startsWith` predicate never matches.

**Fix:** run the command from inside the `horizon` container:

```
docker compose exec -T horizon php artisan horizon:pause-supervisor finngen

   INFO  Sending USR2 signal to process: 26.
```

Verified paused:

```
| 92bf35037e6e-Mw2h:finngen      | 26  | paused  | redis:finngen (1)               | auto      |
```

Runbook update required: RUNBOOK.md Steps 2 + 5 must invoke `horizon` container, not `php` container, for supervisor control.

## Step 4: Run migration (scoped via `--path=`)

### Step 4.0 — initial attempt (failed)

```
docker compose exec -T php php artisan migrate \
  --path=database/migrations/2026_04_19_000100_isolate_finngen_schema.php --force

SQLSTATE[42501]: Insufficient privilege: must be able to SET ROLE "parthenon_owner"
  (CREATE SCHEMA IF NOT EXISTS finngen AUTHORIZATION parthenon_owner)
```

Root cause: default Laravel `pgsql` connection uses `parthenon_app`, which is NOT a member of `parthenon_owner` — cannot `SET ROLE parthenon_owner` for the `AUTHORIZATION` clause.

Laravel transaction rolled back cleanly (finngen schema did not persist, 5161 rows intact).

### Step 4.1 — Rule 3 auto-fix: grant CREATE on DATABASE to parthenon_migrator

Root cause: design intent is that migrations run as `parthenon_migrator` (member of `parthenon_owner`), but `parthenon_migrator` lacked `CREATE ON DATABASE parthenon`. `deploy.sh` lines 470–479 already switches to `parthenon_migrator` via env override for `./deploy.sh --db`; that path must have been broken on this dev DB since the role split was introduced.

Executed one-time admin op via `claude_dev` (host PG17 superuser per `reference_claude_dev_db.md`):

```
psql -U claude_dev -h localhost parthenon -c "GRANT CREATE ON DATABASE parthenon TO parthenon_migrator"
GRANT

psql -U claude_dev ... -Atc "SELECT has_database_privilege('parthenon_migrator', 'parthenon', 'CREATE')"
t
```

### Step 4.2 — retry as parthenon_migrator (failed again)

Now the migration got past Block 1 (CREATE SCHEMA) but failed at Block 6 (INSERT SELECT):

```
SQLSTATE[42501]: permission denied for table cohort_definitions
  (INSERT INTO finngen.endpoint_definitions ... SELECT ... FROM app.cohort_definitions)
```

`parthenon_migrator` had no grants on `app.cohort_definitions`. The table was owned by `parthenon_app` (14 of 229 `app.*` tables; most are owned by `parthenon_owner`).

### Step 4.3 — Rule 3 auto-fix: transfer ownership to parthenon_migrator

`ALTER TABLE ... DROP COLUMN` requires ownership (not just grants); transferring to `parthenon_migrator` aligns with the 8 existing FinnGen-related tables already owned by `parthenon_migrator`:

```
psql -U claude_dev ... -c "ALTER TABLE app.cohort_definitions OWNER TO parthenon_migrator"
ALTER TABLE
```

### Step 4.4 — Rule 1 auto-fix: restore parthenon_app DML grants

The ownership transfer dropped the prior implicit owner's rights:

```
before: parthenon_app=arwdDxtm/parthenon_app  (implicit, via ownership)
after:  <none>                                 (owner lost implicit rights)
```

Runtime would break. Re-granted immediately:

```
GRANT SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER
  ON TABLE app.cohort_definitions TO parthenon_app
```

Final ACL: `parthenon_app=arwdDxt/parthenon_migrator` — all DML privileges present (`m` = MAINTAIN is the only lost bit; does not affect Laravel).
