## Decisions Made

- **Model stub is defensive-by-default** — included `HasFactory` trait, full PHPDoc `@property` block, and `$keyType='string'` even though Wave 0 doesn't exercise them, so Wave 3 can extend without revisiting type scaffolding.
- **No CoverageBucket helper beyond `label()` + `allValues()`** — CoverageProfile set the precedent; added `label()` with human-readable display strings for the 5 buckets so the frontend can consume enum cases without duplicating string tables.
- **Factory uses `T?_???????` name pattern** — matches the FinnGen endpoint naming convention (letter + digit + underscore + 7 letters, e.g., `I9_AF`, `C3_COLON_ADENO`); `strtoupper()` ensures uniqueness is case-insensitive.
- **`finngen_ro` password fallback chain** — `DB_FINNGEN_RO_PASSWORD` (new) → `FINNGEN_PG_RO_PASSWORD` (existing) → `''` (empty). Co-locates the secret with FinnGenSourceContextBuilder's existing consumer (RESEARCH.md §Reusable Assets).
- **Plan text said "6 FinnGen tables move" but IsolationMigrationTest asserts 7** — The 6 existing `app.finngen_*` tables plus the new `finngen.endpoint_definitions` yields 7 rows in `information_schema.tables WHERE table_schema='finngen'` after Wave 2. VALIDATION.md SC 2 lists all 7 explicitly, so the test is correct. Documented here so Plan 02 planner/executor doesn't get confused.

## Issues Encountered

**Worktree Docker context mismatch** — The PHP container mounts the main repo's `backend/` at `/var/www/html`, not the worktree's `backend/`. Running Pint/PHPStan inside the container would not have seen the worktree edits.

**Resolution:** Symlinked the main repo's `backend/vendor/` into the worktree as `backend/vendor`, then ran host-side Pint 1.29.0 and PHPStan 2.1.40 (same binaries the CI container uses) against the worktree's files directly. Output matches the Dockerized CI toolchain exactly. Orchestrator can remove the symlink during merge if desired (it was never committed — `vendor/` is gitignored).

**Docker services not required for Wave 0** — Plan 01 is purely scaffolding; Docker services are needed starting in Wave 2 (running the migration) and especially Wave 5 (cutover). No need to spin up the stack.
