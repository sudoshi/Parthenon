## 1. What Is Poseidon

Poseidon is a **new Parthenon service** that extends the Aqueduct data ingestion pipeline with repeatable, incremental, dependency-aware OMOP CDM refreshes. It uses:

- **dbt (Data Build Tool)** — SQL-based transformation layer. Each OMOP CDM table is a dbt model with incremental materialization, schema tests, and source freshness checks.
- **Dagster** — Orchestration layer. Schedules, monitors, and coordinates dbt runs. Software-Defined Assets map 1:1 to dbt models. Supports per-source cron schedules, event-driven sensors, and manual triggers.

**Why it exists:** Aqueduct handles one-time bulk ETL loads. But healthcare data sources (EHR, LIMS, PACS, claims) produce data continuously. Poseidon keeps the CDM current with incremental loads, idempotent transformations, dependency-aware execution, and per-source scheduling.

**Poseidon does NOT replace Aqueduct** — it extends it. Aqueduct still handles file upload → profiling → concept mapping → schema mapping. Poseidon picks up after mappings are confirmed and handles the repeatable T (Transform) in ELT.

### 2.3 Key Architecture Decisions Made

| Decision | Value | Why |
|----------|-------|-----|
| dbt packages.yml version format | Array notation `[">=x", "<y"]` | dbt semver parser rejects pip-style comma-strings |
| Dockerfile user | `poseidon` (non-root via addgroup/adduser --system) | HIGHSEC spec compliance |
| dbt project_dir resolution | `pathlib.Path(__file__).parent.parent / "dbt"` in dbt_assets.py | Container-portable, no hardcoded paths |
| Dagster run storage | PostgreSQL via dagster.yaml (generated by entrypoint.sh) | No SQLite in Docker; uses `DATABASE_URL` env var |
| dbt profile host | `host.docker.internal` (env-overridable) | Connects to host PG17, not Docker PG16 |
| dbt profile user | `claude_dev` (env-overridable) | Superuser on host PG17 via ~/.pgpass |
| CDM model materialization | `view` (default in dbt_project.yml) | Phase 1 validation only; will change to `incremental` in Phase 2 |
| Dagster webserver port | 3100 (via `POSEIDON_PORT` env var) | Avoids collision with existing services |
| Build context | `.` (project root) | Dockerfile needs access to `poseidon/` directory |

## 4. Critical Context for Continuing Agents

### 4.1 Database Architecture

- **Single `parthenon` database**, schema-isolated. Vocabulary lives in shared `vocab` schema.
- **Host PG17** on port 5432, NOT Docker PG16. Use `host.docker.internal` from containers.
- **`claude_dev` user** is superuser on host PG17 (credentials via `~/.pgpass`).
- **CDM schemas per source:** `omop` (Acumenus), `synpuf` (SynPUF), `irsf` (IRSF), `pancreas`, `inpatient`, `eunomia`
- **Results schemas:** `results`, `synpuf_results`, `irsf_results`, `pancreas_results`
- dbt writes to CDM schemas. **NEVER write to `vocab` schema** (read-only).

### 4.2 Environment Variables

The Poseidon containers need these env vars (set in docker-compose.yml):

| Variable | Value | Purpose |
|----------|-------|---------|
| `DAGSTER_HOME` | `/app/storage` | Dagster storage directory |
| `DATABASE_URL` | `postgresql://claude_dev:${HECATE_PG_PASSWORD}@host.docker.internal:5432/parthenon` | Dagster run storage + dbt connection |
| `DBT_PROFILES_DIR` | `/app/poseidon/dbt` | Where dbt finds profiles.yml |
| `POSEIDON_PG_HOST` | `host.docker.internal` | dbt profile host (env_var in profiles.yml) |
| `POSEIDON_PG_USER` | `claude_dev` | dbt profile user |
| `POSEIDON_PG_PASSWORD` | `${HECATE_PG_PASSWORD}` | dbt profile password |
| `PARTHENON_API_URL` | `http://nginx:80/api/v1` | Laravel API base for callbacks |
| `REDIS_URL` | `redis://:${REDIS_PASSWORD}@redis:6379/0` | Status broadcasting |

### 4.3 HIGHSEC Rules

- All Dockerfiles MUST have non-root users
- All API routes MUST have `auth:sanctum` + permission middleware
- All Eloquent models MUST use `$fillable` (never `$guarded = []`)
- Webhook endpoint MUST validate `POSEIDON_WEBHOOK_SECRET`
- No unauthenticated routes serving clinical data
- See `.claude/rules/HIGHSEC.spec.md` for full spec

### 4.4 Existing Aqueduct Services to Interface With

These Laravel services exist and Poseidon Phase 4 must integrate with them:

- `EtlSqlGeneratorService` — Generates SQL for CDM writes from confirmed mappings
- `EtlTableMapping` / `EtlFieldMapping` models — Schema mapping records
- `ConceptMapping` model — AI/human-reviewed concept mappings
- `SourceProfile` / `FieldProfile` models — Source data profiling results
- `WriteCdmDataJob` — Current one-shot CDM writer (Poseidon replaces this for recurring feeds)
- `RunValidationJob` — Post-load validation (Poseidon's dbt tests + quality models supplement this)

### 4.7 Known Issues / Watch-outs

1. **dbt manifest:** Dagster needs a compiled dbt manifest at startup. On first run, `dbt parse` or `dbt compile` must run inside the container before Dagster can discover assets. The `@dbt_assets(manifest=dbt_project.manifest_path)` decorator will fail if no manifest exists. May need to add `dbt parse` to entrypoint.sh.

2. **Docker build context:** The Dockerfile uses `.` (project root) as build context, so `COPY poseidon/` works. If you change the build context, all COPY paths break.

3. **dbt packages.yml:** Uses array version notation `[">=x", "<y"]`, NOT comma-strings. dbt's semver parser rejects the comma format.

4. **dagster-dbt version compatibility:** dagster-dbt 0.25.x must match dagster 1.9.x. If upgrading dagster, upgrade dagster-dbt in lockstep.

5. **Person model is a view:** The current `person.sql` is materialized as a `view` (pass-through from `omop.person`). Phase 2 should change CDM models to `incremental` with `merge` strategy for production use.

6. **No `.env` changes:** Only `.env.example` was modified. The actual `HECATE_PG_PASSWORD`, `REDIS_PASSWORD`, and `POSEIDON_WEBHOOK_SECRET` must exist in `backend/.env` for containers to start. The user may need to add them.
