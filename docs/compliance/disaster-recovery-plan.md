## 1. Purpose

This plan establishes procedures for recovering the Parthenon outcomes research platform from events that disrupt normal operations, ranging from single-container failures to full server loss. It defines recovery objectives, backup inventories, and step-by-step restoration procedures for every critical component.

---

## 2. Recovery Objectives

| Metric | Target | Rationale |
|---|---|---|
| **Recovery Point Objective (RPO)** | 24 hours | Daily automated backups at 3:17 AM. Maximum data loss is one day's work. |
| **Recovery Time Objective (RTO)** | 4 hours | Time to restore from backup and bring all services online for a full database corruption scenario. |
| **Maximum Tolerable Downtime (MTD)** | 8 hours | Research operations can tolerate up to a full workday of downtime without critical impact. |

## 3. Backup Inventory

### 3.1 Database Backups

| Backup Script | Schedule | Schemas Covered | Retention | Location |
|---|---|---|---|---|
| `scripts/db-backup.sh` | Daily 3:17 AM (cron) | `app`, `irsf`, `irsf_results`, `vocab`, `results`, `omop`, `public` (Orthanc) | 30 daily backups | `backups/` directory |
| `scripts/pg-host-basebackup.sh` | Weekly (manual/cron) | Full cluster (WAL-level) | 4 weeks | `backups/` directory |
| `scripts/pg-host-logical-backup.sh` | On demand | Per-schema logical dumps | Manual | `backups/` directory |
| `scripts/pg-host-prune-backups.sh` | Weekly | N/A (pruning) | N/A | Removes backups beyond retention |
| `scripts/backup-app-db.sh` | On demand | Application tables only | Manual | `backups/` directory |

**Backup features:**
- Timestamped filenames: `parthenon-full-YYYYMMDD-HHMMSS.sql`
- `latest.sql` symlink always points to most recent backup
- Integrity manifest in JSON: row counts for `app.users`, `app.sources`, `app.cohort_definitions`, `app.characterizations`, `app.analysis_executions`, `irsf.person`, `vocab.concept`, `results.cohort`
- pg_dump flags: `--clean --if-exists --no-owner --no-acl`

### 3.2 Infrastructure Backups

| Component | Backup Method | Schedule | Location |
|---|---|---|---|
| Wazuh configuration | `scripts/wazuh-backup.sh` | Weekly Sunday 4:00 AM (cron) | `backups/` directory |
| Application code | Git (GitHub remote) | Every commit | GitHub repository |
| Docker images | Dockerfile rebuild from repo | On demand | Defined in `docker-compose.yml` |
| Environment secrets (`.env`) | Encrypted manual copy | On every credential change | Offsite encrypted storage |
| LUKS keyfile | Encrypted manual copy | On creation/rotation | Offsite encrypted storage (separate from `.env`) |
| Solr indexes | Rebuilt from database | On demand | Regenerated by re-indexing |
| ChromaDB embeddings | Rebuilt from source documents | On demand | Regenerated by `ingest.py` |
| Qdrant vectors | Rebuilt from source data | On demand | Regenerated from embeddings pipeline |

### 3.3 What Is NOT Backed Up (Regenerable)

These components are derived data that can be regenerated:
- Solr search indexes (10 configsets) — rebuilt by re-indexing from database
- ChromaDB embeddings — rebuilt by `python3 ~/.parthenon-brain/ingest.py`
- Qdrant vectors — rebuilt from embeddings pipeline
- Redis cache — ephemeral by design; rebuilds on application access
- Frontend build artifacts (`frontend/dist/`) — rebuilt by `npx vite build`
- Achilles/DQD results — regenerated by re-running characterization jobs

## 4. Recovery Scenarios

### Scenario A: Single Container Failure

**Impact:** One Docker service is unhealthy or stopped. Other services continue operating.
**RTO:** 5–15 minutes.

```bash
# 1. Identify the failed service
docker compose ps

# 2. Check logs for the failure reason
docker compose logs --tail 100 <service>

# 3. Attempt restart
docker compose restart <service>

# 4. If restart fails, rebuild and recreate
docker compose build <service>
docker compose up -d <service>

# 5. Verify health
docker compose ps
# For PHP/API: curl https://parthenon.acumenus.net/api/v1/health
```

**Common container-specific notes:**
- **parthenon-darkstar (R runtime):** Takes ~60 seconds to start due to HADES package loading. Wait before declaring failure.
- **parthenon-horizon:** If queue worker dies, jobs accumulate in Redis. Restart picks up pending jobs automatically.
- **parthenon-solr:** After restart, verify cores loaded: `curl http://localhost:8983/solr/admin/cores?action=STATUS`
- **parthenon-postgres:** If PostgreSQL restarts, all dependent services (PHP, AI, R) will reconnect automatically via connection pooling. Check for in-flight transactions that may have been lost.

### Scenario B: Database Corruption

**Impact:** PostgreSQL data is corrupted, inconsistent, or partially lost. Application may return errors.
**RTO:** 1–2 hours.

```bash
# 1. Stop the application layer (preserve DB for diagnosis)
docker compose stop php nginx horizon reverb

# 2. Assess the damage
docker compose exec postgres psql -U parthenon -c "\dt app.*"
docker compose exec postgres psql -U parthenon -c "SELECT count(*) FROM app.users;"
docker compose exec postgres psql -U parthenon -c "SELECT count(*) FROM vocab.concept;"
# Compare against backups/latest.json integrity manifest

# 3. Identify the last clean backup
ls -lt backups/*.sql | head -10
cat backups/latest.json  # Check integrity counts

# 4. If corruption is limited to specific schemas, restore selectively:
#    Extract and restore just the affected schema from the full dump
#    (Requires manual editing of the SQL dump or using pg_restore with -n flag)

# 5. For full corruption, restore the entire database:
docker compose stop postgres
# Clear the corrupted data volume
docker compose rm -v postgres
docker compose up -d postgres
# Wait for PostgreSQL to initialize
sleep 10
# Restore from backup
docker compose exec -T postgres psql -U parthenon < backups/latest.sql

# 6. Restart application services
docker compose up -d

# 7. Verify
curl https://parthenon.acumenus.net/api/v1/health
# Spot-check row counts against backup manifest
```

### Scenario C: Full Server Loss

**Impact:** Production server is completely unavailable (hardware failure, hosting provider issue, catastrophic compromise).
**RTO:** 4 hours (assuming a replacement server is available).

**Prerequisites:**
- Access to a new Ubuntu 24.04 server
- Access to GitHub repository
- Access to offsite backup copy (database dump + `.env` + LUKS keyfile)

```bash
# 1. Provision new Ubuntu 24.04 LTS server
# Install prerequisites
sudo apt update && sudo apt install -y git docker.io docker-compose-plugin python3 python3-pip

# 2. Clone the repository
git clone <repo-url> ~/Github/Parthenon
cd ~/Github/Parthenon

# 3. Run the installer
python3 install.py --with-infrastructure
# This sets up Docker services, creates database schemas, configures Solr, etc.

# 4. Restore secrets
# Copy .env from offsite encrypted backup to backend/.env
chmod 600 backend/.env

# 5. Restore LUKS encryption (if data volume is separate)
sudo cryptsetup luksFormat /dev/<data-device>
sudo cryptsetup luksOpen /dev/<data-device> md0_crypt
# Restore LUKS keyfile from offsite backup
sudo cp /path/to/backup/.luks-keyfile /root/.luks-keyfile
sudo chmod 600 /root/.luks-keyfile
sudo cryptsetup luksAddKey /dev/<data-device> /root/.luks-keyfile

# 6. Restore database
docker compose exec -T postgres psql -U parthenon < /path/to/backup/latest.sql

# 7. Verify integrity
docker compose exec postgres psql -U parthenon -c "SELECT count(*) FROM app.users;"
docker compose exec postgres psql -U parthenon -c "SELECT count(*) FROM vocab.concept;"
# Compare against backup manifest JSON

# 8. Rebuild search indexes
# Solr: re-index vocabularies and cohorts
# ChromaDB: re-ingest documentation
python3 ~/.parthenon-brain/ingest.py -s ~/Github/Parthenon --include-code

# 9. Configure DNS
# Update parthenon.acumenus.net A record to point to new server IP

# 10. Configure SSL
sudo apt install certbot python3-certbot-apache
sudo certbot --apache -d parthenon.acumenus.net

# 11. Run full deployment
./deploy.sh

# 12. Verify all services
docker compose ps
curl https://parthenon.acumenus.net/api/v1/health
```

### Scenario D: Ransomware

**Impact:** Files or database encrypted by ransomware. System is compromised.
**RTO:** 4–8 hours (includes forensic preservation).

```bash
# 1. IMMEDIATELY isolate the server
# Contact hosting provider to disconnect network (preserve disk state)
# Do NOT attempt to boot or access the system normally

# 2. If possible, snapshot the compromised disk for forensic analysis

# 3. Follow Scenario C (Full Server Loss) on a CLEAN server
# Do NOT restore from backups onto the compromised server

# 4. Before restoring backups, verify they are clean:
# - Check backup file timestamps (are they from before the compromise?)
# - Scan backup files with antivirus
# - Restore to an isolated environment first and verify data integrity

# 5. After recovery, conduct full security review:
# - How did the ransomware enter? (Wazuh logs from before compromise)
# - What vulnerabilities were exploited?
# - Were credentials compromised? (Rotate ALL credentials)
# - Was ePHI exfiltrated before encryption? (Check for data breach)
```

### Scenario E: Accidental Data Deletion

**Impact:** A schema, table, or set of records was accidentally dropped or deleted.
**RTO:** 30 minutes – 1 hour.

```bash
# 1. Stop writes to prevent further damage
docker compose stop horizon  # Stop queue processing

# 2. Identify what was lost
docker compose exec postgres psql -U parthenon -c "\dt <schema>.*"

# 3. Restore from latest backup (selective)
# Extract the relevant schema from the full dump and replay
# Or restore the full backup to a temporary database and copy the missing data:
docker compose exec postgres createdb -U parthenon parthenon_recovery
docker compose exec -T postgres psql -U parthenon parthenon_recovery < backups/latest.sql
docker compose exec postgres pg_dump -U parthenon -n <schema> parthenon_recovery | \
  docker compose exec -T postgres psql -U parthenon parthenon

# 4. Clean up
docker compose exec postgres dropdb -U parthenon parthenon_recovery

# 5. Restart services
docker compose up -d
```

## 5. Offsite Backup Procedure

To meet HIPAA requirements for data availability, critical backups must be stored offsite:

### 5.1 What Goes Offsite

| Item | Frequency | Method |
|---|---|---|
| Database backup (`latest.sql`) | Weekly | Encrypted copy to offsite storage |
| Environment secrets (`backend/.env`) | On every change | Encrypted copy to offsite storage |
| LUKS keyfile | On creation/rotation | Encrypted copy to separate offsite location |
| Wazuh backup | Monthly | Encrypted copy to offsite storage |

### 5.2 Encryption for Offsite Transport

```bash
# Encrypt a backup for offsite storage
gpg --symmetric --cipher-algo AES256 backups/latest.sql
# Produces backups/latest.sql.gpg

# Decrypt when needed
gpg --decrypt backups/latest.sql.gpg > backups/restored.sql
```

Store the GPG passphrase separately from the encrypted backups.

## 6. Recovery Testing Schedule

| Test | Frequency | Procedure | Success Criteria |
|---|---|---|---|
| **Backup file verification** | Monthly | Verify backup files exist, check timestamps, validate integrity JSON | Files present, timestamps current, row counts non-zero |
| **Full restore test** | Quarterly | Restore `latest.sql` to a temporary database, compare row counts | All schemas restored, row counts match manifest within 1% |
| **Container rebuild test** | Semi-annually | `docker compose build --no-cache && docker compose up -d` | All 29 containers healthy within 10 minutes |
| **Full DR drill** | Annually | Simulate server loss: provision new server, restore from backups, bring online | Full stack operational within RTO (4 hours) |

### 6.1 Restore Test Procedure

```bash
# 1. Create test database
sudo -u postgres createdb parthenon_restore_test

# 2. Restore
sudo -u postgres psql parthenon_restore_test < backups/latest.sql 2>&1 | tee backups/restore-test.log

# 3. Verify row counts
sudo -u postgres psql parthenon_restore_test -c "SELECT count(*) FROM app.users;"
sudo -u postgres psql parthenon_restore_test -c "SELECT count(*) FROM vocab.concept;"
sudo -u postgres psql parthenon_restore_test -c "SELECT count(*) FROM omop.person;"
sudo -u postgres psql parthenon_restore_test -c "SELECT count(*) FROM irsf.person;"
# Compare against backups/latest.json

# 4. Document results
cat >> backups/restore-verification.log <<EOF
=== Restore Test $(date) ===
Backup file: $(ls -la backups/latest.sql)
Status: [PASS/FAIL]
Row counts: [matching/discrepancies]
Duration: [time]
Tester: Sanjay Mudoshi
EOF

# 5. Clean up
sudo -u postgres dropdb parthenon_restore_test
```
