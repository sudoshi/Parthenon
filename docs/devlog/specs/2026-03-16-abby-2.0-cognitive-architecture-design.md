# Abby 2.0 — Cognitive Architecture Design Spec

**Date:** 2026-03-16
**Status:** Approved
**Author:** Collaborative design (user + Claude)

## Overview

Abby 2.0 transforms Parthenon's AI research assistant from a RAG-enabled Q&A chatbot into a stateful cognitive agent with layered memory, hybrid intelligence, supervised autonomy, and institutional learning. Designed for large-scale, multi-site research institutions with hundreds of users across departments.

### Design Principles

- **Hybrid LLM:** Local models (MedGemma) for routine tasks, cloud API (Claude) for complex reasoning, with strict data governance
- **Supervised autonomy:** Abby proposes action plans, users confirm before execution
- **Open knowledge sharing:** All institutional knowledge visible to all researchers
- **Meticulous phased delivery:** Full vision designed upfront, built in careful phases where each ships production-hardened

### Current State (Abby 1.0)

- MedGemma 1.5 4B via Ollama (fast but limited reasoning)
- ChromaDB RAG across 6 collections (docs, per-user conversations, shared FAQ, clinical reference, OHDSI papers, unified)
- 8 live database context tools (concept sets, cohorts, vocabulary, Achilles stats, DQD, CDM summary, analyses)
- 18 page-context-specific system prompts
- Per-user conversation memory with 90-day TTL
- Streaming SSE responses through Laravel proxy
- Basic user profile (name + roles) passed with each request

## Section 1: Four-Tier Memory System

The cognitive backbone of Abby 2.0 — how she remembers, learns, and personalizes.

### Tier 1: Working Memory (Session Context)

What Abby holds in her head during a single conversation.

- **Current conversation turns** (already exists)
- **Active page context** with entity details (already exists, upgraded):
  - If user is on a cohort page, Abby pre-loads the cohort definition, criteria, patient count, generation status, and recent modifications
  - If user is on a vocabulary page, she loads the concept hierarchy, relationships, and usage statistics
- **Inferred intent stack** — tracks what the user is trying to accomplish across multiple turns, not just the latest message. Example: user asks "what's the diabetes prevalence?" then "break it down by age" — Abby knows the second question is about diabetes, not a fresh query
  - **Stack rules:** Max depth of 3 topics. Explicit topic changes ("now let's look at hypertension") clear the stack. Implicit shifts detected by low semantic similarity to current stack top trigger a confirmation ("Are we still talking about diabetes, or switching to hypertension?"). Stack entries expire after 10 turns of inactivity on that topic.
- **Scratch pad** — intermediate reasoning artifacts (SQL drafts, cohort specs, analysis parameters) that persist within the session for iterative refinement
- **Capacity management** — sliding window with summarization. When context exceeds the LLM's effective window, older turns are compressed into summaries rather than dropped
  - **Summarization model:** MedGemma handles summarization (cost-free, fast). Trigger: when conversation exceeds 70% of the active model's context window. Summaries are tagged as synthetic and the user can expand to see original turns if needed. Summarization prompt is deterministic and tested against a golden set of 20+ conversation patterns.

### Tier 2: Episodic Memory (Per-User History)

What Abby remembers about each individual researcher across sessions.

- **Conversation archive** — upgrades current 90-day TTL ChromaDB collection to permanent, searchable archive in PostgreSQL with vector embeddings
- **User research profile** (new `abby_user_profiles` table):
  - Research interests and active projects (learned, not configured)
  - Expertise level per domain (auto-calibrated from question complexity and vocabulary). **Transparency:** Profile is visible and editable by the user via a "My Research Profile" settings page. Calibration uses exponential decay weighting (recent interactions count more). Users can reset any domain score. An expert asking a basic question does not downgrade their profile — calibration requires sustained pattern over 5+ interactions.
  - Preferred interaction style (verbose explanations vs. terse answers, detected from feedback signals)
  - Frequently used concept sets, cohorts, and vocabularies
  - Common workflows and shortcuts
- **Learning loop:** After each conversation, Abby extracts and stores:
  - Key facts the user shared ("I'm studying incident diabetes in elderly populations")
  - Preferences expressed ("just give me the SQL, I don't need the explanation")
  - Corrections made (if user says "no, I meant Type 2 specifically" — Abby remembers this disambiguation for next time)

### Tier 3: Semantic Memory (Domain Knowledge)

Abby's deep understanding of the clinical research domain.

- **Existing RAG collections** (docs, clinical_reference, ohdsi_papers) — retained and enhanced
- **OMOP CDM knowledge graph** (new): Structured representation of concept relationships, not just flat embeddings. Abby understands that "metformin" is a drug used for "Type 2 diabetes mellitus" which is a subtype of "diabetes mellitus" — enabling inference, not just keyword matching
  - **Storage:** Leverages existing OMOP `concept_relationship` and `concept_ancestor` PostgreSQL tables (already contain millions of rows). No separate graph database. Add materialized views for common traversal patterns (ancestor paths, sibling concepts, cross-domain relationships) with proper indexing. Add a Redis cache layer for hot paths (top 1000 most-queried concepts). Refresh materialized views on vocabulary update (triggered by ETL completion).
  - **Query interface:** A `KnowledgeGraphService` class exposing: `get_ancestors(concept_id, levels)`, `get_descendants(concept_id, levels)`, `find_related(concept_id, relationship_types)`, `find_path(concept_a, concept_b)`, `get_siblings(concept_id)`. Uses recursive CTEs against the materialized views.
- **Institutional data profile:** Abby maintains a living understanding of the specific CDM:
  - Which vocabularies have the best coverage
  - Which domains have sparse data (so she can warn researchers)
  - Temporal coverage (date ranges per data source)
  - Known data quality issues from DQD results
- **Research methodology knowledge:** Best practices for cohort design, study protocols, common pitfalls (immortal time bias, confounding, etc.) — sourced from OHDSI guidelines and enriched over time

### Tier 4: Institutional Memory (Cross-User Intelligence)

What Abby learns from the entire research community using Parthenon.

- **Discovery ledger:** When a researcher finds something noteworthy (a useful cohort design, an unexpected data quality issue, a novel concept set combination), it gets captured and made available to others
- **Pattern recognition:** Abby notices when multiple researchers are working on similar topics and can surface relevant work ("Dr. Chen built a similar hypertension cohort last month — would you like to see her approach?")
- **FAQ evolution:** Upgrades the existing `faq_shared` collection from manual promotion to automatic detection — when the same question gets asked 3+ times across users, it becomes an institutional FAQ with a vetted answer
- **Cohort/analysis templates:** Successful cohort definitions and analysis configurations become reusable templates, tagged by disease area, study design, and methodology
- **Institutional knowledge base:** Curated findings, standard operating procedures, and methodological decisions that represent how this institution does research — distinct from generic OHDSI guidance

## Section 2: Intelligence Layer — Hybrid LLM Architecture

### Model Routing Strategy

A lightweight classifier (runs locally) examines every incoming request and routes it to the appropriate model based on task complexity and type.

**Local (MedGemma 4B via Ollama) — handles ~70% of requests:**
- Intent classification and triage
- Simple factual lookups
- Concept synonym resolution and vocabulary navigation
- Embedding generation for memory storage/retrieval
- Thinking token extraction and preprocessing
- Quick data summaries from cached/pre-computed results

**Cloud (Claude via API) — handles ~30% of requests:**
- Multi-step reasoning
- Analysis interpretation
- Cohort critique and methodology review
- Complex natural language to SQL translation
- Action planning and multi-step workflow orchestration
- Synthesizing across multiple memory tiers into a coherent response
- Creative and open-ended research guidance

### Router Implementation

**Two-stage routing: rule-based pre-filter + model classifier**

```
User Message
    |
Stage 1: Rule-Based Pre-Filter (zero-cost, <1ms)
    | Deterministic rules:
    | - Message contains "create", "build", "run", "modify", "delete" → CLOUD (action planning)
    | - Message length > 200 chars with multiple clauses → CLOUD (complex reasoning)
    | - Message references multiple entities or time periods → CLOUD
    | - Message is a simple greeting, thanks, or single-entity lookup → LOCAL
    | - Message matches FAQ cache exactly → LOCAL (cached answer)
    |
    +-- CLOUD (deterministic match)
    +-- LOCAL (deterministic match)
    +-- UNCERTAIN → Stage 2
    |
Stage 2: Lightweight Classifier
    | A fine-tuned distilbert-base model (~66M params, <50ms inference)
    | trained on labeled routing decisions from production traffic
    | Outputs: route (local|cloud) + confidence score
    |
    +-- confidence >= 0.7 → route as classified
    +-- confidence < 0.7 → CLOUD (err toward capability)
    +-- fallback: if MedGemma response gets thumbs-down → auto-escalate to Claude for retry
```

**Key design decisions:**
- The router errs toward Claude when uncertain — better to over-spend than under-deliver on a clinical research question
- Users never see which model answered — Abby is one persona regardless of backend
- Response quality feedback (thumbs up/down) feeds back into both router calibration and the classifier training set
- All cloud API calls are logged with request/response metadata for cost tracking and audit
- **Bootstrap strategy:** Before sufficient labeled data exists to train the classifier, Stage 1 rules handle all routing. Classifier is introduced once 500+ labeled routing decisions are collected from production traffic.

### Model Versioning & Upgrade Strategy

- **Regression test suite:** A labeled evaluation set of 100+ query/expected-route pairs, run before any model swap
- **Router recalibration:** When MedGemma or Claude is upgraded, re-run the evaluation set and adjust confidence thresholds
- **Canary deployment:** New models serve 10% of traffic for 48 hours. Compare thumbs-up rate, latency, and escalation rate against baseline. Promote only if metrics hold or improve.
- **Prompt version pinning:** All system prompts are versioned. Model upgrades do not automatically change prompts — prompt adaptation is a separate, tested step.

### Data Governance for Cloud Calls

**Compliance target:** HIPAA Safe Harbor method. No individual-level clinical data leaves the network boundary.

**Primary defense — architectural boundary:**
- Cloud API calls are assembled by the Context Assembly Pipeline, which has an explicit allowlist of data sources that can be included in cloud-bound prompts
- **Allowed sources (cloud-safe):** User's natural language query, aggregate statistics (counts, percentages, means), concept/vocabulary names and hierarchies, cohort definition structure (criteria logic, not patient lists), analysis configurations, methodology questions, institutional knowledge artifacts, help documentation
- **Blocked sources (never sent to cloud):** Raw SQL query results containing individual records, any data from `cdm.person`, `cdm.visit_occurrence`, or other clinical event tables at the individual level, conversation content flagged as containing patient references, any data from live context tools unless pre-aggregated

**Secondary defense — sanitization layer (defense in depth):**
- A `PHISanitizer` class runs on every cloud-bound prompt before transmission
- **Detection:** Combines regex patterns (SSN, MRN, phone, email, date-of-birth formats) with a lightweight NER model (spaCy `en_core_web_sm` PERSON entity detection) for name detection
- **Action:** Detected PHI is replaced with `[REDACTED]` tokens. The original is never sent. A warning is logged for review.
- **False-negative mitigation:** Quarterly adversarial testing with synthetic PHI-laden prompts (50+ test cases covering direct identifiers, quasi-identifiers, and edge cases like rare disease + demographics combinations)
- **Audit:** Every cloud API call logged with: timestamp, user_id, data classification tier, SHA-256 hash of the sanitized payload, sanitizer findings (redaction count, entity types detected), response hash

**Circuit breaker for PHI incidents:**
- If the sanitizer detects PHI in a cloud-bound prompt, the request is blocked (not just redacted) and a security alert is raised
- Monthly audit of all cloud API logs by a designated compliance reviewer

### Context Assembly Pipeline

```
1. Working Memory    --> current session state, intent stack, scratch pad
2. Episodic Memory   --> user profile, relevant past conversations (top 3 by similarity)
3. Semantic Memory   --> RAG results, knowledge graph traversal, data profile warnings
4. Institutional     --> relevant discoveries, templates, FAQs from other users
5. Live Context      --> real-time database queries triggered by intent detection
6. Page Context      --> current UI state and entity details
        |
   Context Ranker (scores each piece by relevance to current query)
        |
   Token Budget Allocator (fits within model context window, prioritizes by score)
        |
   Assembled Prompt --> LLM
```

The context ranker prevents the "everything but the kitchen sink" problem — instead of dumping all available context into the prompt, each piece is scored for relevance and the budget allocator ensures the most important context gets priority within the token window.

**Token budget profiles:**

| Tier | MedGemma (4K effective) | Claude (128K effective) | Priority |
|------|------------------------|------------------------|----------|
| Working Memory | 1500 tokens | 8000 tokens | Highest — always included |
| Page Context | 500 tokens | 2000 tokens | High — always included |
| Live Context | 800 tokens | 4000 tokens | High — included when tools triggered |
| Episodic Memory | 400 tokens | 4000 tokens | Medium — top 3 by similarity |
| Semantic Memory | 600 tokens | 6000 tokens | Medium — top 5 RAG results |
| Institutional | 200 tokens | 4000 tokens | Lower — top 3 artifacts |

**Guaranteed minimums:** Data quality warnings (Tier 3) and safety-critical context always receive at least 200 tokens regardless of ranking. If a data quality warning exists for the domain being discussed, it cannot be truncated.

**Phase 1 note:** Context assembly initially targets MedGemma only. Phase 2 parameterizes the pipeline by model, adding the Claude budget profile and cloud-safety filtering.

### Streaming & Response Quality

- **Streaming maintained** for all responses (SSE through Laravel proxy — existing pattern)
- **Confidence indicators:** Abby signals when she's uncertain
- **Source attribution upgraded:** Every claim links back to its source — which memory tier, which document, which database query result
- **Self-consistency check:** For high-stakes responses (cohort design, methodology advice), Claude runs a second pass to verify its own reasoning before delivering

## Section 3: Agency — Plan-Confirm-Execute Framework

### Action Taxonomy

Every action classified by risk level, which determines the confirmation UX:

**Low Risk (inline confirm — single "Go ahead" button):**
- Search vocabulary for concepts
- Look up cohort/concept set details
- Run pre-built reports and summaries
- Export data to CSV/clipboard
- Save a conversation as a note

**Medium Risk (plan preview — user reviews step-by-step before confirming):**
- Create a new concept set from discussed concepts
- Build a cohort definition from natural language spec
- Queue a characterization or incidence analysis
- Modify an existing cohort's inclusion/exclusion criteria
- Generate SQL for data exploration

**High Risk (detailed plan + explicit confirmation per step):**
- Execute generated SQL against the CDM
- Run a full study package (multiple analyses)
- Modify or delete existing cohorts/concept sets
- Bulk operations affecting multiple entities

### The Plan-Confirm-Execute Loop

```
User request
    |
PLAN PHASE -- Abby decomposes into steps with specific parameters
    |
CONFIRM PHASE -- Presented as reviewable card with [Approve All] [Edit Plan] [Cancel]
    |
EXECUTE PHASE -- Step-by-step with real-time status; pauses on failure with alternatives
    |
REPORT PHASE -- Summary of completed actions with links to created entities
```

### Tool Registry

Expands the current 8 live context tools into a full action toolkit. Each tool declares:

```json
{
  "name": "tool_name",
  "description": "What it does",
  "risk_level": "low | medium | high",
  "parameters": {},
  "requires_confirmation": true,
  "rollback_capable": true
}
```

**New tools beyond current live context:**

| Tool | Risk | Description |
|------|------|-------------|
| `create_concept_set` | medium | Build concept set from concept IDs |
| `create_cohort_definition` | medium | Define cohort with entry/inclusion/exclusion |
| `generate_cohort` | medium | Execute cohort generation, return count |
| `run_characterization` | medium | Queue characterization analysis |
| `run_incidence_analysis` | medium | Queue incidence rate analysis |
| `export_results` | low | Export analysis results to file |
| `compare_cohorts` | low | Side-by-side comparison of two cohorts |
| `clone_cohort` | low | Duplicate an existing cohort as starting point |
| `modify_concept_set` | high | Add/remove concepts from existing set |
| `modify_cohort_criteria` | high | Change criteria on existing cohort |
| `execute_sql` | high | Run validated read-only SQL against CDM |
| `schedule_recurring_analysis` | high | Set up periodic re-execution |

### Concurrency Control

Multi-user environments require protection against conflicting concurrent actions:

- **Optimistic locking:** Every entity that Abby can modify (cohort definitions, concept sets, analyses) carries a `version` column (integer, incremented on every write). When Abby generates a plan, the plan captures the current version of each target entity. At execution time, each step verifies the version hasn't changed. If it has, the step fails with a clear message: "This cohort was modified by another user since you approved this plan. Please review the current state and try again."
- **Plan expiry:** Approved plans expire after 10 minutes. If the user doesn't confirm within that window, Abby re-validates the plan against current state before executing.
- **Entity-level locking for high-risk actions:** `modify_concept_set` and `modify_cohort_criteria` acquire an advisory lock (PostgreSQL `pg_advisory_lock`) on the entity ID for the duration of execution. If another action holds the lock, the second action waits up to 5 seconds, then fails with a message indicating who holds the lock.

### Rollback & Safety

**Rollback strategy by tool category:**

| Tool | Rollback Type | Mechanism | Time Limit |
|------|--------------|-----------|------------|
| `create_concept_set` | True rollback | Delete created entity (blocked if referenced by a cohort) | 1 hour |
| `create_cohort_definition` | True rollback | Delete created entity (blocked if generation has run) | 1 hour |
| `generate_cohort` | Compensating action | Mark generation as invalidated; does not delete results but flags them | 24 hours |
| `modify_concept_set` | True rollback | Restore previous concept list from checkpoint | 1 hour |
| `modify_cohort_criteria` | True rollback | Restore previous criteria JSON from checkpoint | 1 hour (blocked if generation ran against new criteria) |
| `clone_cohort` | True rollback | Delete the clone | 1 hour |
| `run_characterization` | Compensating action | Cancel if queued; mark results as withdrawn if completed | Until completion |
| `run_incidence_analysis` | Compensating action | Cancel if queued; mark results as withdrawn if completed | Until completion |
| `execute_sql` | Not rollback-capable | Read-only queries produce no state change | N/A |
| `schedule_recurring_analysis` | True rollback | Delete the schedule entry | Indefinite |

**Downstream dependency checks:** Before rollback, Abby checks if any other entities were created that depend on the entity being rolled back. If dependencies exist, Abby lists them and asks the user to confirm cascade rollback or manual resolution.

**Checkpoint storage:** `abby_action_log.checkpoint_data` stores the full previous state of the modified entity as JSONB. For create actions, checkpoint_data is null (rollback = delete). For modify actions, checkpoint_data contains the complete prior entity state.

- **Dry run mode:** For high-risk actions, simulate the action and report expected outcome before execution. Dry run queries use `BEGIN; ... ROLLBACK;` transaction pattern.
- **Rate limiting:** Configurable per tool type. Defaults: Low-risk tools — 20/min, 200/hour. Medium-risk tools — 10/min, 50/hour. High-risk tools — 3/min, 20/hour. DAG-internal steps (within an approved multi-step plan) count as a single action against the rate limit.
- **Audit trail:** Every action logged with user, timestamp, plan, parameters, result, checkpoint_data, and rollback state. Retention: 12 months in primary table, then archived to `abby_action_log_archive` (same schema, partitioned by month).

### Cost Controls & Circuit Breaker (Cloud API)

- **Budget model:** Global monthly pool with per-department soft quotas. Tracked in a `abby_cloud_usage` table: `(id, user_id, department, tokens_in, tokens_out, cost_usd, model, created_at)`.
- **Alerting thresholds:** Automated alerts at 50%, 80%, and 95% of monthly budget. Alerts go to system administrators via existing notification channels.
- **Circuit breaker:** At 95% budget, cloud routing is disabled. All requests fall back to MedGemma with a user-visible banner: "Cloud AI is temporarily unavailable due to usage limits. Responses may be less detailed for complex questions." Administrators can override.
- **Per-user visibility:** Users see their cloud usage in the "My Research Profile" page (no hard per-user limits by default, but configurable).
- **Degraded mode:** When cloud is unavailable (budget exhaustion, API outage, network issue), Abby transparently falls back to local-only with a confidence caveat on complex queries: "I answered this locally — for a more thorough analysis, try again when cloud AI is available."

### Multi-Step Workflow Orchestration

For complex requests spanning multiple tools, Claude decomposes into a directed acyclic graph (DAG) of tool calls. Independent steps run in parallel. User sees and approves the full plan before execution begins.

## Section 4: Institutional Knowledge & Collaboration

### Knowledge Capture Pipeline

Automatic capture triggers:

| Event | What's Captured | Storage |
|-------|----------------|---------|
| Cohort created successfully | Definition pattern, concept set composition, design rationale | `abby_knowledge_artifacts` |
| Analysis completed | Configuration, key findings summary, interpretation notes | `abby_knowledge_artifacts` |
| User corrects Abby | Misconception and correction as learning pair | `abby_corrections` |
| Same question asked 3+ times | Question + vetted answer promoted to institutional FAQ | `faq_shared` |
| Novel concept set combination | Unusual but successful concept groupings | `abby_knowledge_artifacts` |
| Data quality issue discovered | Finding, affected tables/domains, workaround | `abby_data_findings` |

**Knowledge Artifact Structure:**
```json
{
  "type": "cohort_pattern | analysis_config | methodology | data_finding | faq",
  "title": "Incident T2DM cohort with washout period",
  "summary": "...",
  "tags": ["diabetes", "incident", "new_users", "washout"],
  "disease_area": "endocrinology",
  "study_design": "incident",
  "created_by": "user_id",
  "usage_count": 0,
  "embedding": [],
  "artifact_data": {}
}
```

### Knowledge Surfacing

Contextual suggestions at the right moment:
- Starting a cohort similar to existing ones → offer as starting point
- Exploring a domain with known data gaps → warn proactively
- Creating a concept set that overlaps with an existing one → suggest reuse

### Collaboration Features

- **Shared artifacts:** Publish cohorts, concept sets, analyses to institutional library with versioning and usage tracking
- **Research threads:** Abby can connect researchers working on similar topics (with consent)
- **Institutional Q&A:** Researchers flag useful responses for promotion to institutional knowledge

### Knowledge Quality Management

- **Staleness detection:** Artifacts unused for 6+ months flagged for review
- **Conflict resolution:** Contradicting artifacts flagged for senior researcher resolution
- **Accuracy scoring:** Track artifact success rate over time
- **Deprecation:** Vocabulary updates or data refreshes trigger re-validation of affected artifacts

## Phased Delivery Plan

### Phase 1: Memory Foundation

**Goal:** Abby remembers who you are and what you're working on.

**Delivers:**
- `abby_user_profiles` table and learning loop
- Working memory upgrade — intent stack and scratch pad
- Conversation archive upgraded to permanent PostgreSQL-backed storage with vector search
- Context assembly pipeline — ranked, budget-aware prompt construction (MedGemma profile only; designed for extension in Phase 2)

**Migration plan (ChromaDB → PostgreSQL):**
- Backfill: Export all existing ChromaDB conversation collections to PostgreSQL `abby_messages` with embeddings. Run as a one-time migration script.
- Dual-read period (2 weeks): New writes go to PostgreSQL. Reads check PostgreSQL first, fall back to ChromaDB for older data not yet migrated.
- Cutover: Once backfill verified complete (row count + spot-check), disable ChromaDB reads. ChromaDB retained read-only for 30 days as safety net.
- Embedding compatibility: Current `all-MiniLM-L6-v2` embeddings are preserved as-is. If embedding model changes in a future phase, a re-embedding migration script will be provided.
- Rollback: If PostgreSQL-backed memory degrades retrieval quality (measured by A/B comparison of retrieval relevance scores), revert to ChromaDB reads within the dual-read window.

**Success criteria:**
- Returning users: Thumbs-up rate on responses improves by 15%+ for users with 3+ prior sessions vs. new users (measured via existing feedback system)
- Multi-turn coherence: Intent stack correctly maintains topic across 20+ turns in a test suite of 10 scripted conversation scenarios
- Latency: p95 response time stays under 3s for local responses (measured via Grafana dashboard)
- Migration: Zero data loss verified by row count comparison between ChromaDB export and PostgreSQL import

### Phase 2: Intelligence Upgrade

**Goal:** Abby gives meaningfully better answers to hard questions.

**Delivers:**
- Hybrid LLM routing — MedGemma for fast/simple, Claude API for complex reasoning
- Data governance layer — PHI sanitization, cloud call audit trail, data classification
- Confidence indicators and enhanced source attribution
- Router calibration feedback loop

**Success criteria:**
- User satisfaction: Thumbs-up rate on complex queries (those routed to Claude) is 30%+ higher than current MedGemma-only thumbs-up rate. Baseline measured in Phase 1.
- Cloud API costs: Monthly spend stays within administrator-configured budget (tracked in `abby_cloud_usage` table, visible in Grafana)
- PHI protection: Zero PHI detected in monthly cloud API audit log review. Adversarial test suite of 50+ synthetic PHI prompts achieves 100% detection rate.
- Routing accuracy: A labeled evaluation set of 100+ queries achieves > 90% agreement between router decision and human-labeled ground truth. Evaluated before launch and monthly thereafter.

### Phase 3: Semantic Knowledge Graph

**Goal:** Abby understands clinical concepts relationally.

**Delivers:**
- OMOP concept relationship graph materialized for fast traversal
- Institutional data profile — CDM strengths, gaps, temporal coverage
- Proactive data quality warnings
- Enhanced vocabulary navigation with hierarchy traversal

**Success criteria:**
- Concept hierarchy questions answered with 95%+ accuracy
- Data gap warnings fire appropriately
- Concept set suggestions leverage the graph

### Phase 4: Agency Framework

**Goal:** Abby can take actions, not just talk.

**Delivers:**

### Cross-Cutting Concerns (All Phases)

- **Testing:** 80%+ coverage with unit, integration, and E2E tests per phase
- **Monitoring:** Latency, routing accuracy, cloud costs, action success rates — all in Grafana
- **Migration:** Backward-compatible; Abby 1.0 data carries forward
- **Documentation:** Dev blog post per phase, updated help docs
- **Data retention:** `abby_action_log` entries archived after 12 months (partitioned by month). `abby_cloud_usage` retained for 24 months for cost analysis. All new tables included in existing PostgreSQL backup strategy (daily snapshots, WAL archiving).
- **Disaster recovery:** Knowledge artifacts and user profiles are recoverable from the most recent daily backup. Action log loss is acceptable (audit trail, not operational data). Conversation archive loss triggers re-import from ChromaDB backup (retained for 90 days post-migration).
