## Notes

1. **This is not exhaustive for all 186 repos.** The ohdsi-studies GitHub organization contains 186 public repositories. Many are older, completed, or represent sub-packages of larger studies. The entries above capture the actively maintained and most significant studies as of early 2026.

2. **Canonical study listing**: OHDSI maintains an auto-generated study directory at [data.ohdsi.org/OhdsiStudies](https://data.ohdsi.org/OhdsiStudies/) which is populated from structured README metadata in each repository.

3. **2025 Guideline-Driven Evidence (GDE) studies**: In January 2025, OHDSI launched 14 new guideline-driven evidence opportunities for community network studies during Phenotype Phebruary. These are actively recruiting sites.

4. **2023 SOS Challenge studies**: Several active studies originated from the 2023 Study-a-thon on-site challenge, including the Semaglutide/NAION study led by Cindy Cai.

5. **DARWIN EU studies**: The European Medicines Agency's DARWIN EU coordination center runs its own OMOP-based characterization studies, which are related to but organizationally separate from the ohdsi-studies GitHub repos.

6. **Older study archives**: Completed studies from the pre-ohdsi-studies era can be found at [github.com/OHDSI/StudyProtocols](https://github.com/OHDSI/StudyProtocols) and [github.com/OHDSI/StudyProtocolSandbox](https://github.com/OHDSI/StudyProtocolSandbox).

**Note**

What's Different About the Newest Studies (2025+)
The 2025 Guideline-Driven Evidence studies and anything using the newer Strategus framework represent a meaningful evolution. Strategus is OHDSI's newer execution engine that replaces the ad-hoc R package approach with a more structured, JSON-driven orchestration layer. If you look at the StrategusStudyRepoTemplate, the study specification is defined as a set of Strategus modules (CohortGenerator, CohortDiagnostics, Characterization, CohortMethod, PatientLevelPrediction, etc.) with all parameters serialized to JSON. This means the study lead defines the complete analysis specification once, and participating sites execute it through a uniform Strategus interface rather than custom R scripts.
The practical difference for a new site is that instead of editing a bespoke codeToRun.R with study-specific parameters scattered throughout, you configure a single set of execution settings (connection details, CDM schema, output folder) and Strategus handles orchestrating all the analytical modules in the correct sequence. It's still R-based and still runs locally, but it's more standardized.
The 2025 GDE studies specifically came out of Phenotype Phebruary, where 14 clinical guideline-driven evidence opportunities were presented in January. The workflow for these is to develop concept sets and cohort definitions collaboratively during February, then package them as Strategus studies for network execution. Sites that want to participate are expected to join the relevant OHDSI workgroup calls and express interest through the Forums.
Where the Pain Points Are (and Where Parthenon Fits)
The fundamental problems a new site faces today:
No self-service onboarding. There's no way to say "I have an OMOP CDM at my institution, show me what studies I can join and let me check if my data supports them." You have to manually discover studies, manually contact leads, manually assess feasibility.
No feasibility pre-check. Before committing to a study, you'd ideally want to know "do I even have enough patients matching the target cohort in my CDM?" Today you'd have to install the study package and run CohortDiagnostics yourself to find out. There's no lightweight feasibility query you can run first.
No centralized status tracking. The study lead has no dashboard showing "Site A completed IRB, Site B is running CohortDiagnostics, Site C submitted results." This is tracked in spreadsheets, Teams channels, and memory.
No governance workflow. IRB submissions, approvals, DUA signatures — all tracked manually if at all.
No result validation pipeline. When a site submits results, the study lead manually inspects the ZIP, loads CSVs into their R environment, and runs EvidenceSynthesis. There's no automated integrity checking or standardized ingestion.
For Parthenon, this maps directly to the Studies module architecture we built in the last conversation. The StudySites management page with its status workflow (invited → onboarding → irb_pending → irb_approved → executing → results_submitted → completed) is exactly what's missing. The feasibility check endpoint that dispatches lightweight cohort count queries through the R Plumber bridge before a site commits to full execution — that doesn't exist in Atlas or any current OHDSI tool. And the study discovery aspect of the StudiesIndex page, where a new site can browse active studies filtered by study type and see "this study needs 3 more sites" — that's currently done through forum posts and community calls.
The single most impactful feature Parthenon could offer a new OHDSI participant is a "Check My Data" workflow: point it at your CDM, it runs a lightweight feasibility query for every active study's target cohorts, and returns a dashboard showing "your database has estimated populations of N for these 8 studies — click to express interest."
