# Roles & Permissions

Parthenon uses role-based access control (RBAC) powered by [Spatie Laravel Permission](https://spatie.be/docs/laravel-permission/). Roles are collections of named permissions assigned to users. A user can hold multiple roles, and their effective permissions are the union of all permissions from all assigned roles.

## Built-in Roles

| Role | Description |
|------|-------------|
| `super-admin` | All permissions. Manages system configuration, authentication providers, and all other admin functions. |
| `admin` | User management, data source management, system health. Cannot change system-level authentication settings. |
| `researcher` | Full read/write on cohorts, concept sets, analyses. Can generate cohorts and run analyses. Cannot manage users or sources. |
| `viewer` | Read-only access to cohort definitions, concept sets, and analysis results. Cannot generate cohorts or run analyses. |

## Permission Reference

Permissions follow the convention `action:resource`:

| Permission | Description |
|-----------|-------------|
| `view:users` | List and view user accounts |
| `manage:users` | Create, edit, suspend, delete users |
| `view:sources` | List data sources |
| `manage:sources` | Create, edit, delete data sources |
| `view:cohorts` | View cohort definitions |
| `manage:cohorts` | Create, edit, delete cohort definitions |
| `generate:cohorts` | Execute cohort generation jobs |
| `view:analyses` | View analysis results |
| `manage:analyses` | Create, edit, delete analyses |
| `run:analyses` | Execute analysis jobs |
| `view:patients` | Access patient-level timeline data (PHI) |
| `manage:system` | System health, queue management, configuration |

## Creating Custom Roles

Administrators can create custom roles for specialized use cases:

1. Navigate to **Admin → Roles**.
2. Click **New Role**.
3. Enter a **Role Name** (snake_case, e.g. `data_steward`).
4. Select the permissions to include.
5. Click **Save**.

Custom roles can then be assigned to users on the User Management page.

## Role Assignment

Users can hold multiple roles. For example, a user might be both a `researcher` (for analytical work) and a `data_steward` (custom role with `manage:sources` permission for source configuration).

Role assignments take effect immediately — the user's next API call uses the updated permission set.

## Super-Admin Bypass

Users with the `super-admin` role bypass all permission checks — they have access to all resources and actions regardless of their explicit permission list. Super-admin status is set in the database and cannot be self-assigned.

:::warning Principle of least privilege
Assign the minimum role required for each user's job function. Avoid granting `researcher` to users who only need to review existing results — use `viewer` instead. This limits the blast radius of compromised credentials.
:::
