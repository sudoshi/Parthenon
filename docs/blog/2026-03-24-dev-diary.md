## The MBU Data Disappearance — Third Time's the Charm (For Real This Time)

If you've been following the devlog, you know the story: Dr. Udoshi's (person_id=1005788) Precision Medicine and Imaging data has now vanished from the Patient Profile **three times**. The culprit was always the same — the original data was inserted via ad-hoc SQL with no repeatable seeder, meaning any database reset, migration run, or accidental truncation wiped it permanently with no way to recover it short of manually re-running those one-off queries.

Today we closed that loop for good.

### What Was Restored

The restoration covers two data domains:

**Genomic Variants** — Four variants from the FoundationOne CDx report (FMI Case TRF091836, dated 2015-05-23) sourced from a colon adenocarcinoma specimen collected during a sigmoid colectomy at UPenn in March 2012:

| Gene | Alteration | Significance | Notes |
|------|-----------|--------------|-------|
| KRAS | G12D | Pathogenic | Resistance to Cetuximab/Panitumumab; potential Trametinib sensitivity |
| TP53 | R282W | Pathogenic | — |
| APC | S1281* | Pathogenic | — |
| SETD2 | Exon 12 rearrangement | Likely pathogenic | — |

**DICOM Imaging Studies** — 5,013 DICOM files reorganized into 15 studies (CT, CR, MR, PET), 117 series, and 5,013 instances spanning 2013–2015, imported from `docs/MBU/DICOM/`. The import required temporarily mounting that directory into the PHP container — that volume mount has been reverted in `docker-compose.yml` post-import.

### The Fix: `mbu:seed-genomics` Artisan Command

The heart of the fix is the new `backend/app/Console/Commands/SeedMbuPatient.php` command, registered as `mbu:seed-genomics`. The command is fully **idempotent** — it checks for existing records before inserting, so it's safe to run after any database reset without worrying about duplicates. Going forward, this should be part of the standard post-reset / post-migration runbook for any environment standing up demo data.

## Onboarding Tour Migrated to React Joyride v3

We bumped `react-joyride` from 2.9.3 to 3.0.0, which brought a breaking API change to how the tour lifecycle is managed. The `OnboardingModal` component was updated in `d35b4a216` to align with the v3 callback and prop signatures — if you're referencing the old `continuous` + `run` pattern from v2, that's now changed.

A secondary fix (`a97951eca`) corrected the `WhatsNewModal`, which was incorrectly using `window.open` to manage open/close state instead of the component's own `isOpen` React state. That was causing the modal to behave unpredictably on re-render. Both fixes are now live and the onboarding tour flows correctly end-to-end.

## Orthanc Data Volume Now Configurable

A small but meaningful infrastructure improvement: the Orthanc DICOM server's data volume path is now configurable via the `ORTHANC_DATA_PATH` environment variable (`b665cc770`). Previously it was hardcoded, which made it awkward to point at a larger or external storage volume in non-default environments. Set it in your `.env` and the compose file picks it up — no more editing `docker-compose.yml` directly for this.

## What's Next
